# 解决AI代理技术利用率52%的完整架构方案

**核心发现：** AI代理在BMAD/SuperPower工作流中跳过技术能力审计，导致技术栈平均仅52%利用率——这是一个可通过"强制审计门控"模式系统性解决的架构缺陷。本报告提供五大维度的完整生产级解决方案：BMAD Story模板改造、CLAUDE.md强制提问配置、Graphiti记忆追踪、斜杠命令体系、以及16代理并行修复架构。所有代码块可直接复制使用。

## Part 1: BMAD Story模板——技术能力审计扩展

BMAD V6的Story模板（`template.md`）在`Technical Implementation Notes`和`Tasks`之间缺少一个关键环节：**技术能力发现与确认**。以下模板扩展在BMAD V6的步骤文件架构（step-file architecture）中增加了`step-02b-tech-audit.md`，作为分析（step-02-analyze）和草稿（step-03-draft）之间的强制门控。

### 1.1 新增步骤文件：`step-02b-tech-audit.md`

```markdown
# Step 02b: Technology Capability Audit

## Purpose
Before drafting tasks, the SM agent MUST audit every technology mentioned in the
Architecture document and this Epic to discover ALL available capabilities — not
just the ones implied by the user story.

## Instructions

### Phase 1 — Technology Inventory
1. Read `docs/architecture.md` and the parent Epic file.
2. List every technology, framework, library, and service mentioned.
3. For EACH technology, use Context7 MCP to fetch current documentation:
   - Call `resolve-library-id` with the technology name
   - Call `query-docs` with the resolved ID and query "all features and capabilities"

### Phase 2 — Capability Enumeration
For each technology, produce a numbered capability list:

```
### [Technology Name] — Capability Register
Source: Context7 library ID /org/repo | version X.Y.Z
Last audited: {{current_date}}

1. [Capability Name] — [One-line description] — Status: ⬜ NOT EVALUATED
2. [Capability Name] — [One-line description] — Status: ⬜ NOT EVALUATED
...N. [Capability Name] — [One-line description] — Status: ⬜ NOT EVALUATED
```

### Phase 3 — User Confirmation (MANDATORY GATE)
Present the full capability list to the user and ask:
> "I found N capabilities for [Technology]. Which of these do you need for
>  this story? Please respond with the numbers, or 'all', or 'skip'."

**DO NOT proceed to Step 03 until the user has responded for EVERY technology.**

### Phase 4 — Status Assignment
Based on user response, update each capability status:
- ✅ SELECTED — Will be implemented in this story
- ⏳ PENDING — Selected but deferred to a future story (add to Pending Register)
- ⬜ NOT NEEDED — Explicitly declined by user
- ❌ NOT AVAILABLE — Capability exists but blocked by constraints

### Phase 5 — Store in Graphiti
If Graphiti MCP is available, store each capability decision:
- Call `add_episode` with the capability decisions as structured JSON
- Use group_id matching the project name
```

### 2.3 Hook脚本实现

**`.claude/hooks/inject-tech-audit-context.sh`**

```bash
#!/bin/bash
# Fires on every user prompt — injects audit reminder into context
INPUT=$(cat)
PROMPT=$(echo "$INPUT" | jq -r '.prompt // empty')

# Check if prompt mentions implementation-related keywords
if echo "$PROMPT" | grep -qiE '(implement|build|create|add|setup|configure|integrate|story|feature)'; then
    cat <<'EOF'
REMINDER: Before implementing, you MUST:
1. Check if a Technology Capability Audit exists for all technologies involved
2. If not, run the audit: use Context7 to fetch docs, enumerate capabilities,
   and ask the user which ones to enable
3. Record decisions in the story's Technology Capability Audit section
4. Only then proceed with implementation

Check for existing audit: grep -r "Capability Register" _bmad-output/implementation-artifacts/stories/
EOF
fi
exit 0
```

**`.claude/hooks/enforce-audit-gate.sh`**

```bash
#!/bin/bash
# Fires before Write/Edit — blocks if writing to src/ without audit
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.filePath // empty')

# Allow writes to non-source files (docs, configs, tests, audit files)
if echo "$FILE_PATH" | grep -qvE '^(src/|app/|lib/|packages/)'; then
    exit 0
fi

# Check if any story is currently in-progress with a completed audit
AUDIT_COMPLETE=$(find "$CLAUDE_PROJECT_DIR/_bmad-output/implementation-artifacts/stories/" \
    -name "*.story.md" -newer /tmp/.claude-session-start 2>/dev/null | \
    xargs grep -l "Audit Status: 🟢 COMPLETE" 2>/dev/null | head -1)

if [ -z "$AUDIT_COMPLETE" ]; then
    # Check for a lighter signal — any capability register in recent conversation
    if [ -f "$CLAUDE_PROJECT_DIR/.claude/audit-approved" ]; then
        exit 0
    fi
    echo "NOTE: No completed Technology Capability Audit found in current stories. Consider running /tech-audit before implementing." >&2
    # Exit 0 (warning only) instead of exit 2 (block) to avoid over-blocking
    # Change to exit 2 for strict enforcement
    exit 0
fi
exit 0
```

**`.claude/hooks/verify-coverage-on-stop.sh`**

```bash
#!/bin/bash
# Fires when Claude stops responding — nudges coverage verification
INPUT=$(cat)
STOP_REASON=$(echo "$INPUT" | jq -r '.stop_reason // empty')

if [ "$STOP_REASON" = "end_turn" ]; then
    # Check for any TODO/FIXME related to capabilities
    PENDING=$(grep -rn "Status: ⏳ PENDING\|ENABLE_.*=false\|TODO.*capability\|# TODO.*inject" \
        "$CLAUDE_PROJECT_DIR/src/" 2>/dev/null | head -5)
    if [ -n "$PENDING" ]; then
        echo "⚠️ COVERAGE WARNING: Found pending capabilities or disabled features:"
        echo "$PENDING"
        echo ""
        echo "Run /coverage-check to verify all selected capabilities are implemented."
    fi
fi
exit 0
```

## Part 3: Graphiti实体类型——技术能力追踪

### 3.1 自定义实体类型定义（Pydantic模型）

```python
"""
graphiti_capability_schema.py
Custom Graphiti entity and edge types for tracking technology capability decisions.
Place in: graphiti/mcp_server/custom_types.py or project's graphiti config.
"""
from pydantic import BaseModel, Field
from typing import Optional
from datetime import datetime


# ============================================================
# ENTITY TYPES (Nodes)
# ============================================================

class TechnologyStack(BaseModel):
    """A technology, framework, library, or service in the project stack."""
    version: Optional[str] = Field(None, description="Current version in use")
    category: Optional[str] = Field(
        None,
        description="Category: language, framework, database, service, tool, runtime"
    )
    total_capabilities: Optional[int] = Field(
        None, description="Total number of known capabilities"
    )
    enabled_capabilities: Optional[int] = Field(
        None, description="Number of capabilities currently enabled"
    )
    utilization_percent: Optional[float] = Field(
        None, description="Percentage of capabilities utilized (enabled/total * 100)"
    )
    context7_id: Optional[str] = Field(
        None, description="Context7 library ID for documentation lookup"
    )
    last_audited: Optional[datetime] = Field(
        None, description="When this technology was last capability-audited"
    )


class TechnologyCapability(BaseModel):
    """A specific capability or feature of a technology that can be enabled."""
    status: Optional[str] = Field(
        None,
        description="Status: SELECTED, PENDING, NOT_NEEDED, NOT_AVAILABLE, ENABLED, DISABLED"
    )
    priority: Optional[str] = Field(
        None, description="Priority: critical, high, medium, low"
    )
    configuration_key: Optional[str] = Field(
        None, description="Environment variable or config key that controls this capability"
    )
    configuration_value: Optional[str] = Field(
        None, description="Current configuration value"
    )
    target_story: Optional[str] = Field(
        None, description="Story ID where this capability will be implemented"
    )
    decided_by: Optional[str] = Field(
        None, description="Who made the decision (user or agent)"
    )
    decided_at: Optional[datetime] = Field(
        None, description="When the capability decision was made"
    )


class CapabilityDebt(BaseModel):
    """A capability that was selected but not yet properly configured or enabled.
    Represents technical debt in the capability coverage."""
    severity: Optional[str] = Field(
        None, description="Severity: critical, high, medium, low"
    )
    debt_type: Optional[str] = Field(
        None,
        description="Type: dead_code, disabled_feature, partial_config, missing_injection, empty_shell"
    )
    file_path: Optional[str] = Field(
        None, description="Primary file where the debt exists"
    )
    remediation_steps: Optional[str] = Field(
        None, description="Steps needed to resolve this debt"
    )
    estimated_effort: Optional[str] = Field(
        None, description="Estimated effort: trivial, small, medium, large"
    )
    blocking_story: Optional[str] = Field(
        None, description="Story ID that is blocked by this debt"
    )


class AuditReport(BaseModel):
    """A technology utilization audit report for the project."""
    overall_utilization: Optional[float] = Field(
        None, description="Average utilization percentage across all technologies"
    )
    total_technologies: Optional[int] = Field(
        None, description="Number of technologies audited"
    )
    total_debt_items: Optional[int] = Field(
        None, description="Number of capability debt items found"
    )
    audit_date: Optional[datetime] = Field(None, description="When the audit was conducted")
    auditor: Optional[str] = Field(None, description="Who conducted the audit")


class ProjectStory(BaseModel):
    """A BMAD story in the project backlog."""
    status: Optional[str] = Field(
        None,
        description="Status: drafted, ready-for-dev, in-progress, review, validated, done"
    )
    epic_id: Optional[str] = Field(None, description="Parent epic identifier")
    audit_status: Optional[str] = Field(
        None, description="Tech audit status: NOT_STARTED, IN_PROGRESS, COMPLETE"
    )


# ============================================================
# EDGE TYPES (Relationships)
# ============================================================

class HasCapability(BaseModel):
    """A technology has a specific capability available."""
    documentation_url: Optional[str] = Field(
        None, description="URL to capability documentation"
    )
    since_version: Optional[str] = Field(
        None, description="Version when this capability was introduced"
    )


class HasDebt(BaseModel):
    """A technology or capability has associated technical debt."""
    discovered_at: Optional[datetime] = Field(
        None, description="When the debt was discovered"
    )
    discovery_method: Optional[str] = Field(
        None, description="How it was found: audit, code_review, runtime_error"
    )


class ImplementedIn(BaseModel):
    """A capability is implemented in a specific story."""
    implementation_date: Optional[datetime] = Field(
        None, description="When the implementation was completed"
    )
    coverage_percent: Optional[float] = Field(
        None, description="How completely the capability was implemented"
    )


class DependsOn(BaseModel):
    """One capability depends on another being enabled first."""
    dependency_type: Optional[str] = Field(
        None, description="Type: hard (must have), soft (nice to have)"
    )
    blocking: Optional[bool] = Field(
        None, description="Whether this dependency is currently blocking"
    )


# ============================================================
# TYPE MAPS
# ============================================================

ENTITY_TYPES = {
    "TechnologyStack": TechnologyStack,
    "TechnologyCapability": TechnologyCapability,
    "CapabilityDebt": CapabilityDebt,
    "AuditReport": AuditReport,
    "ProjectStory": ProjectStory,
}

EDGE_TYPES = {
    "HasCapability": HasCapability,
    "HasDebt": HasDebt,
    "ImplementedIn": ImplementedIn,
    "DependsOn": DependsOn,
}

EDGE_TYPE_MAP = {
    ("TechnologyStack", "TechnologyCapability"): ["HasCapability"],
    ("TechnologyStack", "CapabilityDebt"): ["HasDebt"],
    ("TechnologyCapability", "CapabilityDebt"): ["HasDebt"],
    ("TechnologyCapability", "ProjectStory"): ["ImplementedIn"],
    ("TechnologyCapability", "TechnologyCapability"): ["DependsOn"],
    ("AuditReport", "TechnologyStack"): ["HasCapability"],
}
```

### 3.4 能力决策持久化的Cypher查询示例

```cypher
-- 查询所有待处理的能力（跨会话持久化）
MATCH (t:Entity:TechnologyStack)-[:RELATES_TO]->(c:Entity:TechnologyCapability)
WHERE c.attributes CONTAINS '"status": "PENDING"'
RETURN t.name AS technology, c.name AS capability,
       c.attributes AS details
ORDER BY t.name;

-- 查询能力债务概览
MATCH (d:Entity:CapabilityDebt)
WHERE d.attributes CONTAINS '"severity": "critical"'
   OR d.attributes CONTAINS '"severity": "high"'
RETURN d.name, d.summary, d.attributes
ORDER BY d.created_at DESC;

-- 查询各技术的利用率
MATCH (t:Entity:TechnologyStack)
RETURN t.name AS technology,
       t.attributes AS details
ORDER BY t.name;
```

## Part 4: 斜杠命令完整实现

### 4.1 `/tech-audit`命令

**文件路径：** `.claude/commands/tech-audit.md`

```markdown
---
description: Trigger a full technology capability audit for a specific technology
allowed-tools: Read, Grep, Glob, mcp__context7__resolve-library-id, mcp__context7__query-docs, mcp__graphiti__add_episode, mcp__graphiti__search_nodes
---

# Technology Capability Audit

You are performing a comprehensive capability audit for: $ARGUMENTS

## Step 1: Resolve Technology Documentation
1. Use `mcp__context7__resolve-library-id` with libraryName="$ARGUMENTS"
   and query="all features capabilities configuration options"
2. From the results, select the best matching library
3. Use `mcp__context7__query-docs` with the resolved libraryId and
   query="complete list of all features capabilities configuration options API"
   Set tokens to 10000 for comprehensive results

## Step 2: Enumerate ALL Capabilities
From the Context7 documentation, create a COMPLETE numbered list of every
capability, feature, and configuration option available. Include:
- Core features
- Advanced features
- Security features
- Performance optimization features
- Integration capabilities
- Configuration options with non-default values
- Plugin/extension capabilities
- Debugging/monitoring features

Format as:
```
## [Technology Name] vX.Y.Z — Complete Capability Register
Source: Context7 /org/repo
Audited: [today's date]

### Core Capabilities
1. [Name] — [Description] — Config: [key=default_value]
2. ...

### Advanced Capabilities
N. [Name] — [Description] — Config: [key=default_value]
...

### Security Capabilities
M. [Name] — [Description] — Config: [key=default_value]
...

### Performance Capabilities
...
```

## Step 3: Check Current Project Usage
1. Use Grep to search the codebase for imports, configurations, and usage
   patterns related to this technology
2. For each capability found in Step 2, check if it's currently:
   - ✅ ENABLED and actively used
   - 🟡 IMPORTED but not configured/called
   - ❌ NOT PRESENT in the codebase
3. Calculate current utilization: (ENABLED count / Total capabilities) * 100%

## Step 4: Present to User
Present the complete audit with current status and ask:
> "This technology has [N] capabilities. Currently [M] are enabled ([X]% utilization).
> Here are the capabilities NOT yet enabled:
> [list with numbers]
>
> Which of these would you like to enable? Reply with numbers, 'all', or 'none'."

## Step 5: Record Decisions
After user responds:
1. Update the capability register with user decisions
2. If Graphiti MCP is available, store decisions via `add_episode`:
   - Episode name: "Tech Audit: [technology] - [date]"
   - Episode body: JSON with all capability decisions
   - Source: "tech-audit-command"
3. Output a summary suitable for pasting into a BMAD story's
   Technology Capability Audit section

## Step 6: Generate Implementation Tasks
For each SELECTED capability, generate a concrete implementation task:
```
- [ ] Task: Enable [capability name]
      Config: Set [key]=[value] in [file]
      Code: [specific code change needed]
      Test: [how to verify it works]
```
```

### 4.2 `/coverage-check`命令

**文件路径：** `.claude/commands/coverage-check.md`

```markdown
---
description: Generate a feature coverage checklist for all technologies in the project
allowed-tools: Read, Grep, Glob, Bash, mcp__graphiti__search_nodes, mcp__graphiti__search_facts
---

# Feature Coverage Check

Perform a comprehensive coverage analysis of all technologies in this project.

## Step 1: Identify All Technologies
1. Read the project's architecture document (look in docs/, _bmad-output/, or README)
2. Grep for package.json, Cargo.toml, requirements.txt, go.mod, docker-compose.yml
3. Grep for import statements to identify all frameworks and libraries in use
4. Build a complete technology inventory

## Step 2: For Each Technology, Audit Configuration
For each technology found:

### Configuration Scan
```bash
# Search for feature flags and configuration
grep -rn "ENABLE_\|_ENABLED\|_DISABLED\|feature_flag\|toggle" src/ config/ .env* 2>/dev/null
# Search for TODO/FIXME related to capabilities
grep -rn "TODO.*capability\|TODO.*enable\|TODO.*configure\|FIXME.*feature" src/ 2>/dev/null
# Search for dead code indicators
grep -rn "import.*from\|require(" src/ | sort | uniq -c | sort -rn | head -20
```

### Dead Code Detection
- Imports that are never referenced in the file
- Functions/classes defined but never called
- Tools/plugins declared but never registered/injected
- Feature flags set to false with no toggle mechanism

### Security Configuration Audit
- CSP headers: null or permissive?
- CORS: wildcard (*)?
- Auth tokens: hardcoded or env-based?
- Rate limiting: configured or absent?
- Secret management: plaintext or vault?

## Step 3: Generate Coverage Report

Output format:

## 方案背后的设计原理

**为什么分层防御比单一措施更可靠？** CLAUDE.md中的指令本质上是概率性的——即使用大写MUST，合规率也仅约90%。社区实践和Claude认证架构师考试都明确指出：**当业务规则要求保证合规时，只有程序化执行（Hooks）的失败率为零**。因此本方案将CLAUDE.md作为"教育层"（告诉代理为什么要做审计），Hooks作为"执行层"（确保代理无法跳过审计），Graphiti作为"记忆层"（确保决策跨会话持久化），斜杠命令作为"工具层"（提供标准化操作流程）。

**Graphiti的时间双标模型（bi-temporal model）** 是能力追踪的天然匹配：每个能力决策记录`valid_at`（决策生效时间）和`invalid_at`（决策被替代时间），支持"在Session 3我们决定启用X，但在Session 7改为Y"这样的时间线查询。**`group_id`** 实现项目级命名空间隔离。

**16代理架构采用Wave模式而非全并行**，因为存在数据依赖：Wave 2的Neo4j修复需要Wave 1的Docker/环境配置就位，Wave 3的AI层需要Wave 2的数据层稳定。这与Anthropic在C编译器项目中验证的模式一致——16个代理在Docker容器中运行约2000个会话，通过git-based task locking协调，最终产出10万行编译器代码。

**Context7的两步工作流**（`resolve-library-id` → `query-docs`）被嵌入到每个斜杠命令中，确保代理在实现任何技术前都先阅读最新文档。Context7覆盖9000+库，支持版本特定查询，**索引延迟仅数天**，足以满足大多数生产场景。

本方案中所有模式均源自可验证的生产实践：BMAD V6的步骤文件架构、Claude Code官方Hooks API、Graphiti的自定义Pydantic实体类型、以及Anthropic工程博客中的并行代理编排。将这四个系统整合在一起，就形成了一个完整的"技术能力全生命周期管理"闭环——从发现到决策到实现到验证到跨会话追踪。
