# 用户 Deep Research 提问与回答上下文

> 来源: Claude Code Deep Research session 对话截图 (2026-03-27)

## 用户原始问题

"然后还有一点我现在是使用 Bmad V6或者 SUper power 加Graphiti 开发项目，但是Bmad V6 和Super power 开发代码的质量并不高，开发出来的代码经常有问题要大量返工，然后Graphiti由于积累的记忆过于累赘，导致它在检索的时候并不能找到我的核心决策，它的时序图普不知道为什么没有分清楚哪些是我的重要信息哪些不是，有些决策明明都过期了，所以我需要成熟的SSD开发，我在claude code上的开发返工率太高了；然后Graphiti 是否要在claude code 改成Agentic RAG，请给我提供一些成熟的方案。"

## Claude 确认问题与用户回答

### Q1: Graphiti 记忆质量差的核心症状是哪些？
A: 重要核心决策被淹没在噪音里, 跨 Session 同一概念出现矛盾记忆, 检索结果和当前问题相关性低

### Q2: 你希望 Graphiti 改造的方向是？
A: Graphiti + Agentic RAG 混合架构

### Q3: BMAD V6 / Super Power 代码质量差，你认为根本原因是？
A: Story 粒度太粗，实现太模糊, Agent 没有严格遵守架构约束, 缺乏自动化测试, Bug 无法被及时发现, 提示词没有注入足够的已知错误上下文

### Q4: 你希望解决方案的重点是哪一层？
A: 两者都要，但先给我一个可以注册成 /命令 的完整方案

### Q5:「增量提问不足」的问题，你认为根本原因是？
A: 没有「技术选型审核」这个独立阶段，直接跳到实现, Agent 在开始编码前没有生成「功能覆盖检查清单」并让你逐项确认, Graphiti 里缺少对「技术能力已选但未配置」这类节点的记录, BMAD Story 里没有强制要求 Agent 读 Context7 文档

### Q6: 你最想要的最终输出是什么形式？
A: CLAUDE.md + /commands 模板，直接粘贴使用

## 用户补充反馈

"这里在使用Bmad V6和 Super power加Graphiti 进行开发的时候，就是我是强调了查看Context 7 文档，但是实际开发的时候并没有给我明确具体技术选型的问题，就是我们的技术选型不全，就是我明明说了要实现Graphiti 记忆系统在我自己的开发项目上面，但是在实际实现配套的时候也没有问我，具体要用到哪些功能，所以反而很多的功能都是没有实现，这样技术选型问题普遍存在，然后也不向我进行增量提问"

## 研究重点覆盖

1. SSD高质量实施：Story 原子化、架构约束强制、TDD 集成防返工的成熟方案
2. Graphiti噪音清理与核心决策分层：优先级标记、过期图谱剪枝、矛盾解决机制
3. Graphiti + Agentic RAG 混合架构：具体的路由逻辑、什么时候用图、什么时候用 Agent 动态检索
4. 全部封装成 Claude Code /commands
