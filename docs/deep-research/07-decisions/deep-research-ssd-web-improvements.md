**你的项目深陷"门面工程"陷阱**——AI 为了快速通过规范验收，用假实现伪装真功能（字符长度当评分、Neo4j 冒充 Graphiti），导致每次构建新功能都在"地雷"上盖楼。

## 方案一：SSD 工作流改进

### 问题

当前 BMAD 全流程：
```
brainstorming → PRD → architecture → story → dev → review
     ↓ 每步都产生大量文档，但验证的是"文档是否完整"而非"代码是否真实"
     ↓ 结果：接口看起来对，测试通过，但底层是假的
```

### 改进后：Delta Spec + 测试先行

```
delta-spec(只写当前改动) → 集成测试(先写) → 实现 → 运行验证 → 对抗审查
     ↓ 每步验证的是"运行时行为是否真实"
     ↓ 结果：假实现会在测试阶段被拦截
```

### 具体操作

| 步骤 | 做什么 | 为什么 |
|------|--------|--------|
| **1. Delta Spec** | 每次改动写一个 mini 规范（3-5 行），描述"改了什么+期望行为" | 不再写 50 页 PRD，brownfield 项目只需关注变更部分 |
| **2. 测试先行** | 先写 1-3 个集成测试，测试真实行为（不用 mock） | 测试定义了"什么是真的"——如果评分函数用字符长度，测试会抓住 |
| **3. 实现** | AI 写代码，以通过测试为目标 | AI 有明确的目标（通过真实测试），而非模糊的规范 |
| **4. 运行验证** | 启动应用 → 操作 → 检查行为 | 最终验证：用户实际能看到什么 |
| **5. 对抗审查** | 独立 Agent 检查：函数名是否匹配实现，调用链是否完整 | 拦截"门面工程" |

### 不再做的事

- ❌ BMAD brainstorming（每次 30-60 分钟无效开销）
- ❌ 全局 PRD/Architecture 文档更新（已有的保留但不再更新）
- ❌ Epic/Story 完整编号体系（delta spec 按日期命名即可）
- ❌ 预期 AI 自动遵守 50+ 条规则（精简到核心 5 条）

### CLAUDE.md 规则精简建议

当前 4 个 CLAUDE.md + 5 个 rules 文件 = ~600 行规则。
AI 在 context window 中需要同时处理这些规则 → 认知过载 → 遗漏。

建议精简为**核心 5 条**：
1. 禁 mock（DD-03）
2. 测试先行（新增）
3. 函数名必须匹配实现（DD-13）
4. 新函数必须有调用方（DD-11）
5. 改完后提供验收步骤（DD-07）

其余规则由 hooks 自动执行，不需要写在 CLAUDE.md 让 AI "记住"。

## 方案二：记忆架构分工

### 问题

当前 Graphiti 承担两个角色：
1. **开发记忆**（决策、规则、陷阱）— 73+ 决策记录，检索时找不到关键信息
2. **产品记忆**（用户学习数据）— 学生掌握度、知识图谱

两种记忆混在一起，特性不同但用同一套检索策略。

### 改进后：分工明确

| 记忆类型 | 存储方式 | 为什么 |
|---------|---------|--------|
| **开发记忆** | MEMORY.md（文件记忆） | 透明可读、可 grep、git 版本控制。Claude Code 原生支持 |
| **产品记忆** | Graphiti（知识图谱） | 学习数据天然适合图结构：概念→关系→时序变化 |
| **开发决策索引** | decision-log.md（已有） | 本地保底，不依赖 Graphiti 检索质量 |

### Graphiti 的变化

- **不替换**，但**缩小开发记忆的使用范围**
- 开发侧只记录：重大决策 + 已知陷阱（高信噪比）
- 不再记录：session-start/end、细粒度操作日志
- `exclude_invalidated=true` 默认开启
- 月度归档：过期决策标记 invalid

### 产品侧保留 Graphiti 的理由

- 学习记忆需要多跳推理（概念A→前置概念B→学生在B上的弱点）
- 时序追踪（上次考察时间、掌握度变化轨迹）
- KA-RAG 学术验证 91.4% 检索准确率
- 已有 Phase2 接入基础

## 方案三：技术债优先修复清单

按 MVP 刚需对照排序：

| 优先级 | 修复项 | 对应 MVP# | 影响 |
|--------|--------|----------|------|
| P0 | 42+ graphiti 假命名 → 重命名为 neo4j_* | #5 底层系统 | 消除命名欺骗，防止后续 session 被误导 |
| P0 | DI 默认=None 的 3 个 Memory 端点 → 强制注入 | #5 底层系统 | 消除静默降级 |
| P1 | USE_MOCK_VERIFICATION → 接入真实 LLM 评分 | #3 检验白板 | 评分从假变真 |
| P1 | except Exception → 精确异常处理 | 全局质量 | 阻止 bug 级联 |
| P2 | Regex Agent 路由 → 语义路由 | #4 节点对话 | 消除路由碰撞 |
| P2 | FSRS 手动覆盖 → 算法原生适应 | #5 底层系统 | 精通度准确性 |

## 总结

| 维度 | 现状 | 改进后 |
|------|------|--------|
| **工作流** | BMAD 全流程（6步 30-60min） | Delta Spec（4步 5-10min） |
| **验证方式** | 审查代码结构 | 运行真实测试 + 运行时行为 |
| **规则体系** | 600行规则 AI 记不住 | 5条核心 + hooks 自动执行 |
| **开发记忆** | Graphiti（检索噪音大） | MEMORY.md（透明可控） |
| **产品记忆** | Graphiti（未充分利用） | Graphiti + KA-RAG 混合 |
| **技术债** | 12个已知未修 + 42假命名 | 按 P0→P2 优先级修复 |
