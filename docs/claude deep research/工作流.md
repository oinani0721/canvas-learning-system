# 四大 AI 编码工作流对比与最低返工率混合流程设计

**核心结论：没有任何单一工作流能覆盖 Canvas Learning System 全部任务类型。** 实证数据表明，AI 生成测试的初始变异杀死率仅 **57.3%**，超过 50% 的 AI 测试本质上是实现逻辑的镜像。唯一经过实验验证能将测试级回归率从 6.08% 降至 **1.82%** 的方法是 TDAD 的 AST 依赖图注入——不是告诉 agent「如何做 TDD」，而是告诉它「检查哪些测试」。本报告基于此认知，设计一套分层混合工作流：BMAD V6 负责规划层，GSD 负责上下文工程，Workflow A 物理隔离负责核心执行层，并在 TaskCompleted hook 中嵌入 pytest-gremlins 变异门禁。

## 问题一：PRD → Epic → Story → GSD Task 的完整转换机制

### BMAD V6 的 PRD 拆解为 Epic 应遵循「架构后置」原则

BMAD V6 的一个关键设计决策：Epic 和 Story 在 Phase 3（Solutioning）中创建，**在架构设计完成之后**，而非直接从 PRD 拆解。官方文档明确指出：「Epics and stories are now created after architecture. This produces better quality stories because architecture decisions (database, API patterns, tech stack) directly affect how work should be broken down.」

**PRD → Epic 拆分标准：**

BMAD V6 的 `bmad-create-epics-and-stories` 工作流同时消费 `prd.md` 和 `architecture.md`，按以下维度拆分 Epic：

- **功能模块边界**：每个 Epic 对应一个独立的功能域（如「用户认证」「知识图谱管理」「向量搜索」）
- **架构层对齐**：Epic 必须与 ADR（Architecture Decision Records）中的技术决策一致
- **依赖拓扑**：Epic 之间的依赖关系必须是 DAG（无环图），Epic 1 不应依赖 Epic 3
- **已知问题**：Issue #1688 报告了全量前置展开在大项目中难以管理，建议采用渐进式 Epic 生成

对于 Canvas Learning System，推荐按以下方式拆分：

| Epic | 范围 | 技术栈边界 |
|------|------|-----------|
| E1: 核心数据层 | Neo4j schema + LanceDB 初始化 | FastAPI + Neo4j + LanceDB |
| E2: 知识图谱 CRUD | 节点/关系的增删改查 | FastAPI + Neo4j |
| E3: 向量搜索 | 嵌入生成 + 相似度检索 | FastAPI + LanceDB |
| E4: Tauri IPC 层 | 前后端通信桥接 | Tauri 2 + React |
| E5: React UI | 画布渲染 + 交互逻辑 | React + TypeScript |
| E6: 集成与端到端 | 跨层功能串联 | 全栈 |

### Epic → Story 拆分：1-3 文件原则的实证基础

BMAD V6 文档中**没有明确规定**每个 Story 的文件数上限。但多个来源的实证数据支持 1-3 文件/Story 的经验法则：

- GSD 的 Plan 设计为「fits in ~50% of a fresh context window」，每个 Plan 包含 2-3 个 atomic task，每个 task 通常对应 1-2 个文件
- Workflow A 的 N=1 实测成功案例是在 1-3 文件范围内的纯函数任务
- TDAD 的 test_map.txt 效果在小粒度变更上最显著（直接依赖的测试被准确定位）
- CodeRabbit 2025 分析显示 AI PR 平均大小是人类的 **154%**，大 PR 的缺陷率显著更高

**Story 拆分的具体原则：**

1. **单一职责**：每个 Story 只修改一个「行为单元」——一个 API endpoint、一个 React component、一个 Neo4j 查询模式
2. **接口契约明确**：Story 的输入/输出必须用 TypedDict 或 Pydantic model 定义，写入 `contracts/` 目录
3. **测试可达性**：Story 必须标注测试策略（纯函数/集成/E2E）和预估 mutation 可测性
4. **文件数硬上限**：新增/修改 ≤ 3 个生产文件 + ≤ 2 个测试文件。超过此阈值则必须进一步拆分

### Story → GSD Task 的转换机制

GSD 的 PRD Express Path（`/gsd:plan-phase N --prd path/to/prd.md`）可以直接消费 BMAD Story 文件作为输入。转换流程：

**第一步：Discuss（可选，当 Story 有歧义时启用）**
- GSD 识别 Story 中的「gray areas」——未在 acceptance criteria 中明确定义的产品决策
- 输出 `{phase_num}-CONTEXT.md`，补充 Story 缺失的实现决策
- 对于从 BMAD Story 转换的 Task，可用 `--assumptions` 模式：GSD 读取代码库后列出结构化假设，用户只需确认或修正

**第二步：Plan（核心转换步骤）**
- 3 个协调 agent 并行工作：Researcher（4 个子 agent 探索技术方案）、Planner（拆分为 2-3 个 atomic plan）、Plan-checker（验证 plan 覆盖所有需求）
- 每个 GSD Plan 对应 BMAD Story 的一个子任务，包含精确的文件路径、实现指令、验证命令
- **关键**：Plan 中注入 TDAD 的 `test_map.txt` 相关测试列表，确保 executor 知道检查哪些测试

**第三步：Execute（执行层——此处切换到 Workflow A 或原生 GSD）**
- 对适合物理隔离的任务（纯函数、1-3 文件），切换到 Workflow A 的 tmux 隔离执行
- 对不适合隔离的任务（Tauri IPC、React UI），使用 GSD 原生 executor + Nyquist validation

**第四步：Verify（多层验证）**
- GSD 的 goal-backward verification：「what must be TRUE for this to work?」
- pytest-gremlins 变异测试作为 TaskCompleted hook 的硬门禁
- 跨 phase 回归测试：执行完成后自动运行前置 phase 的测试套件

### 预执行分析的标准化 checklist

每个 Story 执行前必须完成以下分析（推荐用 `/gsd:discuss-phase` 的 assumptions 模式驱动）：

**Story 预执行分析 Checklist：**

1. **TDAD test_map.txt 查询**：`grep "{target_module}" test_map.txt` 获取所有关联测试，列出 top-5 高影响测试
2. **contracts/ 目录 schema 摘要**：运行 `stubgen -p {package} -o contracts/` 生成 `.pyi` 接口文件，确认 Story 涉及的所有 public API 签名
3. **已有测试覆盖分析**：`pytest --cov={module} --cov-report=json` 输出当前覆盖率，标记 <80% 覆盖的函数
4. **依赖变更影响评估**：Story 修改的文件是否被其他模块 import？使用 AST 分析或 `grep -r "from {module}" src/`
5. **技术风险标记**：Neo4j（需要 testcontainers？）、Tauri IPC（不可 unit test？）、第三方 API（需要 mock？）
6. **上下文窗口预算**：预估 Story 需要的上下文 token 数（contracts + test_map + 目标文件 + 测试文件），确保 <50% 窗口

## 问题二：Workflow A 的三个关键未解决问题

### tmux pane-base-index 冲突的精确修复方案

**问题根因**：Claude Code Agent Teams 在 tmux 模式下假定 pane index 从 0 开始。当用户的 `~/.tmux.conf` 设置了 `setw -g pane-base-index 1`（tmux-sensible 插件和大多数教程推荐的配置），Claude Code 的 `send-keys` 指令会发送到错误的 pane，导致 teammate 生成但静止在欢迎界面。此问题记录在 GitHub Issue #23527。

**修复方案（三个层级）：**

```bash
# 方案 A：最小改动（推荐）
# 在项目根目录创建 .claude/tmux.conf，仅在 Agent Teams 会话中覆盖
cat > .claude/tmux-override.sh << 'EOF'
#!/bin/bash
tmux set-option -g base-index 0
tmux setw -g pane-base-index 0
echo "pane-base-index reset to 0 for Agent Teams" >&2
exit 0
EOF
chmod +x .claude/tmux-override.sh

# 在 .claude/settings.json 中用 SessionStart hook 自动执行
# {
#   "hooks": {
#     "SessionStart": [{
#       "hooks": [{
#         "type": "command",
#         "command": "bash .claude/tmux-override.sh"
#       }]
#     }]
#   }
# }
```

```bash
# 方案 B：直接修改 tmux.conf（如果你不介意全局改动）
# ~/.tmux.conf 中注释掉：
# setw -g pane-base-index 1
# 改为：
setw -g pane-base-index 0
```

```bash
# 方案 C：使用 worktree 隔离替代 tmux pane（最稳健）
# 不依赖 tmux pane split，改用 git worktree
git worktree add worktrees/test-writer -b feat/test-writer
git worktree add worktrees/implementer -b feat/implementer
# 每个 agent 在独立 worktree 中运行，通过文件系统完全隔离
```

**附加已知问题**：Issue #23615 报告 Agent Teams 通过 split 当前窗口生成 pane，会破坏用户布局。建议在新 tmux window 中运行 Agent Teams。

### test-writer context 注入的最优组合

test-writer 的上下文应包含三层信息，按优先级排序：

**第一层：contracts/ 目录的 .pyi schema 摘要（必须）**
```bash
# 自动生成并维护
stubgen -p canvas_learning.models -o contracts/
stubgen -p canvas_learning.services -o contracts/
# 生成的 .pyi 文件只包含类型签名，不包含实现
# 例如：
# contracts/canvas_learning/services/graph_service.pyi
# def create_node(label: str, properties: dict[str, Any]) -> NodeId: ...
# def find_related(node_id: NodeId, depth: int = 1) -> list[Relationship]: ...
```

**第二层：TDAD test_map.txt（必须）**
```bash
# 运行 TDAD AST 分析生成
pip install tdad
tdad analyze --repo . --output test_map.txt

# test_map.txt 格式示例：
# src/services/graph_service.py::create_node -> tests/test_graph_service.py::test_create_node_basic (weight: 0.95)
# src/services/graph_service.py::create_node -> tests/test_graph_integration.py::test_create_and_retrieve (weight: 0.72)
```

**第三层：已有测试覆盖率 JSON（推荐）**
```bash
pytest --cov=canvas_learning --cov-report=json -q
# 提取 coverage.json 中目标模块的覆盖率数据
# test-writer 据此知道哪些路径已覆盖、哪些是盲区
```

**注入方式**：通过 PreToolUse hook 限制 test-writer 只能读取 `contracts/`、`test_map.txt`、`coverage.json` 和 `tests/` 目录：

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Read|Glob|Grep",
      "hooks": [{
        "type": "command",
        "command": "bash .claude/hooks/enforce-test-writer-boundary.sh"
      }]
    }]
  }
}
```

```bash
#!/bin/bash
# enforce-test-writer-boundary.sh
INPUT=$(cat)
AGENT=$(echo "$INPUT" | jq -r '.agent_name // ""')
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.pattern // ""')

if [[ "$AGENT" == "test-writer" ]]; then
  # 只允许读取 contracts/, tests/, test_map.txt, coverage.json
  if [[ "$FILE" != contracts/* && "$FILE" != tests/* && \
        "$FILE" != "test_map.txt" && "$FILE" != "coverage.json" && \
        "$FILE" != *.pyi ]]; then
    echo "test-writer 禁止访问实现文件: $FILE" >&2
    exit 2
  fi
fi
exit 0
```

### Claude Code 2026 Agent Teams API 精确用法

**SendMessage 协议**：文件系统级的邮箱通信。每个 agent 有独立的 inbox 文件 `~/.claude/teams/{team-name}/inboxes/{agent-name}.json`。消息信封格式：

```json
[{
  "from": "team-lead",
  "text": "{\"type\":\"task_assignment\",\"taskId\":\"1\",\"description\":\"...\"}",
  "timestamp": "2026-02-18T02:37:16.890Z",
  "read": false
}]
```

已知 Bug（Issue #25135）：SendMessage 在收件人名称不匹配时**静默成功**，消息写入孤立文件无人读取。务必确保 agent name 与 inbox polling name 完全一致。

**TaskCompleted hook**（v2.1.33+）用于变异测试门禁：

```json
{
  "hooks": {
    "TaskCompleted": [{
      "hooks": [{
        "type": "command",
        "command": "bash .claude/hooks/mutation-gate.sh",
        "timeout": 300
      }]
    }]
  }
}
```

```bash
#!/bin/bash
# mutation-gate.sh
INPUT=$(cat)
TASK=$(echo "$INPUT" | jq -r '.task_subject')

# 先跑常规测试
if ! pytest tests/ -x -q 2>&1; then
  echo "常规测试未通过，不允许完成任务: $TASK" >&2
  exit 2
fi

# 再跑变异测试（仅对变更文件）
CHANGED=$(git diff --name-only HEAD~1 -- src/ 2>/dev/null | head -5)
if [ -n "$CHANGED" ]; then
  if ! pytest --gremlins --gremlin-parallel \
       --gremlin-targets="$CHANGED" 2>&1; then
    echo "变异测试发现存活突变体，需要加强测试: $TASK" >&2
    exit 2
  fi
fi
exit 0
```

**TeammateIdle hook**（v2.1.33+）用于动态任务分配：exit code 2 阻止 agent 进入空闲状态并发送新指令。v2.1.64 起支持 JSON decision control `{"continue": false, "stopReason": "..."}`。

## 问题三：四个工作流的严格实证对比

### 对比矩阵

| 维度 | Workflow A（物理隔离） | BMAD V6 Dev Agent | 纯 GSD | Superpowers TDD |
|------|----------------------|-------------------|--------|-----------------|
| **Facade 产生率** | 实测 N=1: 0%（纯函数）；推断：需 testcontainers 时 ~85% 退化 | 实证：Issue #2003 记录多次 facade（「useless assertion on CSS classes」） | 无实测数据；executor 同一 context 写测试+实现，结构上易产生 facade | 无实测数据；Plan 含完整实现代码，subagent context gap（Issue #237） |
| **Mutation kill rate** | 通过 Stop hook 实测 mutmut 通过（N=1） | 无数据 | 无内置变异测试 | 无数据；覆盖率 87-95% 但这是行数覆盖非变异分数 |
| **纯函数任务** | ★★★★★ 最佳场景 | ★★★ 可用但无隔离 | ★★★★ 新鲜上下文有效 | ★★★★ Iron Law 有效 |
| **Neo4j 集成** | ★★ 需 testcontainers 双轨 fixture | ★★ 同上 | ★★★ Nyquist auditor 只读隔离 | ★★ 无特殊处理 |
| **LanceDB 集成** | ★★★★ tmp_path 即可测试 | ★★★ 无隔离优势 | ★★★★ 同上 | ★★★ 同上 |
| **Tauri IPC** | ★ 不可 unit test | ★★ 靠 E2E | ★★ 靠 verify stage | ★★ 靠 E2E |
| **React 前端** | ★★ 物理隔离过度 | ★★★ 可用 | ★★★★ UI review + Playwright | ★★★ 可用 |
| **跨层任务** | ★ 协调成本过高 | ★★★ 单 agent 上下文完整 | ★★★★ 跨 phase 回归门禁 | ★★★ 可用 |
| **Token 消耗** | 高（2 个独立 agent + contracts 维护） | 中（step-file 节省 70-85%） | 高（用户报告 5 小时限额 30 分钟用完） | 中（plan 开销 10-20 min） |
| **失效模式 Top 3** | 1) Neo4j 无内存替代退化 mock  2) 跨层任务协调崩溃  3) 50+文件重构超出单 story 范围 | 1) Dev agent 浅层修复（改名/stub/TODO）  2) Story 模板与验证 checklist 不匹配  3) 数据捏造（超时时编造输出） | 1) 同一 context 的 facade 风险  2) Token 消耗过快  3) 讨论阶段不能消除根本性歧义 | 1) Subagent 缺少纪律上下文（#237）  2) Plan 暴露实现致 facade 风险  3) 不适合快速修复 |

### 关键实证数据点解读

**BMAD V6 的 facade 实证**来自 Issue #2003（2026 年 3 月的回顾），由 BMAD 自己的 agent 进行的回溯分析发现：Story 1.3 中「缺失的 computed style 测试被一个**无用的 CSS class 断言**替代」——这正是 facade test 的典型表现。Issue 作者结论：「没有安全机制迫使 dev agent 重新阅读原始代码、理解问题的本质、或验证修复实际有效。」

**GSD 的 facade 风险是结构性的**但缺乏直接实测。codecentric 的 Thomas Jaspers 明确指出：「Letting the same agent develop and test your application risks one thing: it will no longer fulfill the specification, it will simply learn to pass the tests.」GSD 的 Nyquist auditor（实现文件只读）是部分防御，但常规 execute phase 中 executor 在同一 context 写测试和代码。

**Superpowers 的 facade 风险是推断的**，无直接报告。127K star 的项目中找不到任何 facade 相关 issue。但两个结构性弱点值得关注：(1) Plan 包含完整实现代码的「No Placeholders」规则意味着 subagent 写测试时能看到完整实现；(2) Issue #237 确认 subagent 不接收 `using-superpowers` 纪律上下文。

**TDAD 论文的 TDD 提示悖论**是最有价值的实证洞察：泛泛的 TDD 指令（「先写测试再实现」）将回归率从 **6.08% 恶化到 9.94%**。Django-13089 案例中 TDD 提示导致 352/352 测试全部失败，而 vanilla 仅失败 4 个。核心发现：**agent 不需要被告知如何做 TDD，它们需要被告知检查哪些特定测试。**

## 问题四：最低返工率混合工作流设计

### 总体架构：三层分离

```
┌─────────────────────────────────────────────┐
│           规划层（BMAD V6 Phase 1-3）          │
│  PRD → Architecture → Epic → Story          │
│  输出：story-{slug}.md + contracts/*.pyi     │
├─────────────────────────────────────────────┤
│        上下文工程层（GSD discuss + plan）       │
│  Story → CONTEXT.md → RESEARCH.md → PLAN.md │
│  注入：test_map.txt + coverage.json          │
├─────────────────────────────────────────────┤
│             执行层（任务类型路由）               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ Workflow A │  │ GSD exec  │  │ Manual   │   │
│  │ 物理隔离    │  │ +Nyquist  │  │ +E2E    │   │
│  │ 纯函数/服务 │  │ 前端/UI   │  │ IPC/跨层 │   │
│  └──────────┘  └──────────┘  └──────────┘   │
├─────────────────────────────────────────────┤
│         质量门禁层（全局 hooks）                │
│  TaskCompleted → pytest → pytest-gremlins    │
│  TeammateIdle → 分配下一 story                │
│  跨 phase 回归 → 前置 phase 测试全量运行        │
└─────────────────────────────────────────────┘
```

### 任务类型路由规则

**路由 A → Workflow A 物理隔离**（适用率约 40% 的 Canvas 任务）：
- 纯函数：数据转换、算法逻辑、utility 函数
- 单文件服务：graph_service.py、embedding_service.py 中的独立方法
- 条件：修改 ≤ 3 个文件，无外部依赖或有 tmp_path 替代（如 LanceDB）
- 隔离方式：PreToolUse hook 硬限制 test-writer 只读 `contracts/` + `test_map.txt`

**路由 B → GSD execute + Nyquist**（适用率约 35%）：
- React 组件：UI 渲染逻辑、状态管理
- FastAPI endpoint：请求/响应处理（配合 httpx.AsyncClient 测试）
- 前端-后端接口层：TypeScript 类型定义 + API client
- 优势：新鲜 200K context 避免退化，UI review + Playwright 截图验证

**路由 C → 手动辅助执行**（适用率约 25%）：
- Tauri IPC 层：不可 unit test，需要 E2E 验证
- Neo4j 复杂查询：需要 testcontainers 真实数据库
- 50+ 文件重构：超出 agent 单次能力，需要人工分解和监督
- 跨层集成：涉及 3+ 层的功能串联

### Neo4j 集成层专项处理方案

Neo4j 是 Canvas Learning System 中最大的测试挑战。推荐**三轨策略**：

**第一轨：Repository 模式抽象（架构层解决）**
```python
# contracts/graph_repository.pyi
from abc import ABC, abstractmethod

class GraphRepository(ABC):
    @abstractmethod
    def create_node(self, label: str, props: dict) -> str: ...
    @abstractmethod
    def find_neighbors(self, node_id: str, depth: int) -> list[dict]: ...
    @abstractmethod
    def execute_cypher(self, query: str, params: dict) -> list[dict]: ...
```

这让 test-writer 只需要知道接口契约，不需要知道底层是 Neo4j 还是 NetworkX。

**第二轨：双轨 Fixture（测试层解决）**
```python
# conftest.py
import os, pytest

@pytest.fixture(scope="session")
def graph_repo():
    if os.getenv("USE_TESTCONTAINERS"):
        from testcontainers.neo4j import Neo4jContainer
        container = Neo4jContainer().with_env(
```

### 反馈循环：Story 执行失败时的快速恢复

**失败分类与恢复策略：**

1. **变异测试失败**（最常见）：TaskCompleted hook 拒绝完成 → agent 自动收到 stderr 反馈 → 看到具体存活突变体 → 加强测试断言。**不需要人工干预**，自动重试直到变异门禁通过
2. **回归测试失败**：跨 phase 测试发现破坏 → `git stash` 当前变更 → 运行 `git bisect` 定位引入点 → 用 GSD debugger agent 诊断根因 → 创建 fix plan
3. **facade test 被 mutation 捕获**：变异测试存活率 >30% → 标记当前 Story 为 BLOCKED → 切换到 Workflow A 物理隔离重新执行 RED phase → test-writer 在隔离上下文中重写测试
4. **testcontainers 超时/Docker 不可用**：降级到 NetworkX 路径执行 → 标记 Story 为 PARTIAL → 在 CI 中补跑 integration 测试

```bash
#!/bin/bash
# .claude/scripts/pre-story-analysis.sh
STORY_FILE=$1

echo "=== Story 预执行分析 ==="

# 1. 提取 Story 涉及的文件列表
FILES=$(grep -E "^\s+- (src/|tests/)" "$STORY_FILE" | sed 's/.*- //' | sed 's/ .*//')

# 2. 检查哪些文件已存在
echo "--- 文件状态 ---"
for f in $FILES; do
  if [ -f "$f" ]; then
    LINES=$(wc -l < "$f")
    echo "  EXISTS ($LINES lines): $f"
  else
    echo "  NEW: $f"
  fi
done

# 3. 运行 TDAD 查询关联测试
echo "--- TDAD 关联测试 ---"
for f in $FILES; do
  grep "$f" test_map.txt 2>/dev/null | head -3
done

# 4. 当前测试覆盖率快照
echo "--- 覆盖率快照 ---"
pytest --cov=canvas_learning --cov-report=term-missing -q 2>/dev/null | tail -20

# 5. 最近 5 次相关文件的 git log
echo "--- 最近变更 ---"
for f in $FILES; do
  [ -f "$f" ] && git log --oneline -3 -- "$f" 2>/dev/null
done

# 6. 检查 contracts/ 是否最新
echo "--- contracts/ 状态 ---"
stubgen -p canvas_learning -o /tmp/contracts_check 2>/dev/null
diff -r contracts/ /tmp/contracts_check/ 2>/dev/null && echo "  UP TO DATE" || echo "  STALE - run stubgen to update"
```

## 结论：实证驱动的务实选择

本研究最重要的发现不是某个工作流「最好」，而是**三个被实证数据验证的原则**：

**原则一：告诉 agent 检查哪些测试，而非如何做 TDD。** TDAD 论文证明泛泛 TDD 指令将回归率恶化到 9.94%，而 AST 依赖图注入将其降至 1.82%。这意味着 Workflow A 的 contracts/ + test_map.txt 注入比 Superpowers 的 Iron Law 和 12 条反借口规则更有效——前者提供信息，后者提供纪律。

**原则二：物理隔离对纯函数有效，但覆盖面有限。** Workflow A 的 N=1 实测和 Greenlight 的架构设计都指向同一结论：当任务可以被完全隔离（test-writer 完全不需要看实现）时，facade 率可以降到零。但 Canvas Learning System 中约 60% 的任务（Neo4j 集成、Tauri IPC、跨层功能）无法有效隔离。混合路由是必要的。

**原则三：变异测试是唯一可靠的 facade 检测器。** 覆盖率 87% 的代码可能有 57% 的变异杀死率——行数覆盖不等于行为覆盖。pytest-gremlins 以 **3.73x** 速度优势和 **98% vs 86%** 的检测率优于 mutmut，应作为 TaskCompleted hook 的默认门禁。这个门禁不区分工作流——无论 Workflow A、GSD 还是 Superpowers 产出的代码，都必须通过同一个变异测试关卡。

上述最小可行工作流今天即可部署。它不完美——N=1 的统计局限仍在，contracts/ 维护有成本，PreToolUse hook 的 agent_name 匹配可能需要调试。但它提供了一个**可迭代的起点**：先在下一个纯函数 Story 上运行完整流程，收集第 2 个数据点，然后逐步扩展到集成任务。20 个 Story × 3 次重复的统计显著性目标，从今天开始累积。
