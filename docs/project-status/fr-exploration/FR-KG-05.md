## 5. L1 文本相似度算法（通俗解释）

**一句话**：把文字翻译成数字，比较两串数字的方向是否一致。

### 步骤分解

```
第1步：把文字变成数字（Embedding）

  "线性代数" + 内容描述
         │
         ▼ bge-m3 模型翻译
  [0.12, -0.34, 0.56, 0.08, ..., 0.23]   ← 1024个数字（向量）
                                             代表这段文字的"含义"

  "矩阵运算" + 内容描述
         │
         ▼ bge-m3 模型翻译
  [0.15, -0.31, 0.58, 0.05, ..., 0.21]   ← 含义相近 → 数字接近


第2步：比较两串数字有多像（余弦相似度）

  把两个向量想象成两个箭头：

                      ↗ 向量A（线性代数）
                   θ/
                  /
      原点 ●────────→ 向量B（矩阵运算）

      θ = 两个箭头之间的夹角
      夹角越小 → 方向越一致 → 内容越相似

  计算方式：逐个数字相乘再相加（点积）

  | 夹角 θ | 余弦相似度 | 含义           |
  |--------|-----------|----------------|
  | 0°     | 1.0       | 完全一样        |
  | 53°    | 0.6（阈值）| 够像了，推荐！   |
  | 90°    | 0.0       | 完全无关        |


第3步：保底方案（bge-m3 挂了时）

  退化为 Jaccard 关键词重叠：
  节点A的词：{"线性", "代数", "矩阵", "变换"}
  节点B的词：{"矩阵", "运算", "乘法", "逆"}
  重叠/总共 = 1/7 ≈ 0.14 ← 太低不推荐
  （比 embedding 粗糙很多，只是应急）
```

**关键参数**：
- 模型：BAAI/bge-m3（1024维，中英双语）
- 阈值：0.6（余弦相似度）
- 超时：10s（embedding 调用）
- 源码：`recommendation_service.py:259-347`

## 6. L2 图谱 2 跳算法（通俗解释）

**一句话**：看两个概念的"社交圈"有没有重叠 — 共同好友越多，越应该认识。

### 步骤分解

```
"2跳"是什么意思？

  1跳：A ──边──→ B           （直接连接）
  2跳：A ──边──→ X ──边──→ B  （经过一个中间节点）
  查询 *1..2 = 1跳或2跳都算


实际例子：

   线性代数 ──→ 深度学习 ←── 矩阵运算
                    │
                 共同邻居 #1 ✓

   线性代数 ──→ 统计学 ←── 矩阵运算
                    │
                 共同邻居 #2 ✓

   线性代数          矩阵运算
       ╲                ╱
        没有直接连线！
        但共同邻居 = 2
        → 应该推荐连起来！


置信度怎么算？

  共同邻居数 ÷ 3，最高不超过 1.0

  | 共同邻居数 | 计算        | 置信度 |
  |-----------|-------------|--------|
  | 1         | 1/3         | 0.33   |
  | 2         | 2/3         | 0.67   |
  | 3         | 3/3         | 1.00   |
  | 4+        | min(4/3, 1) | 1.00   |

  为什么除以3？经验值 — 3个以上共同邻居就是很强的信号了。
```

**关键参数**：
- 查询范围：CANVAS_EDGE*1..2（1-2跳）
- 置信度：min(shared/3.0, 1.0)
- a.id < b.id 防止 (A,B) 和 (B,A) 重复
- 源码：`recommendation_service.py:182-222`

## 7. L1 + L2 合并策略

```
L1 发现：线性代数 ↔ 矩阵运算（置信度 0.82，内容相似）
L2 发现：线性代数 ↔ 矩阵运算（置信度 0.67，共同关联2个概念）
L2 发现：概率论 ↔ 统计学     （置信度 0.33，共同关联1个概念）

         │ 同一对？保留置信度更高的
         ▼
合并结果：
  线性代数 ↔ 矩阵运算  → 0.82（L1赢）
  概率论 ↔ 统计学       → 0.33（仅L2）

         │ 去掉用户之前忽略过的
         │ 按置信度从高到低排
         │ 取前5条
         ▼
最终推荐：[线性代数 ↔ 矩阵运算 (0.82), ...]
```

**为什么两层都需要？**

| | L1 擅长 | L1 不擅长 |
|---|---|---|
| **L2 擅长** | 两层都认同 → 超高置信度 | 文字不像但结构上该连 |
| **L2 不擅长** | 孤立节点但内容明显相关 | 都找不到 → 不推荐 |

源码：`recommendation_service.py:357-373`（_merge_candidates）

## 8. Dexie-DB 与忽略记录表

### Dexie-DB 是什么？

浏览器内置数据库 IndexedDB 的易用封装。可以理解为"浏览器里的本地数据库"。

```
┌─────────────────────────────────────────────────────┐
│                浏览器里的存储方式                       │
├──────────────┬──────────┬───────────────────────────┤
│ localStorage │ ~5 MB    │ 只能存字符串，像便签纸      │
├──────────────┼──────────┼───────────────────────────┤
│ IndexedDB    │ 500 MB+  │ 能存对象/数组，像本地数据库  │
│ (Dexie 管的) │          │ 存画布节点/聊天记录/考试状态  │
├──────────────┼──────────┼───────────────────────────┤
│ 后端数据库    │ 无限     │ 真正的服务器数据库          │
│ (Neo4j 等)   │          │ 多设备共享的数据源          │
└──────────────┴──────────┴───────────────────────────┘
```

### 当前 Dexie 有 7 张表（v5）

| 表名 | 存什么 |
|------|--------|
| canvas_boards | 所有白板 |
| canvas_nodes | 白板上的概念节点 |
| canvas_edges | 节点之间的连线 |
| sync_outbox | 待同步队列（离线→上线自动发） |
| chat_messages | 每个节点的聊天记录 |
| crash_recovery | 崩溃恢复用的最后消息 |
| exam_sessions | 考试状态 |

### ❌ 缺失的表：dismissed_recommendations

**后果**：用户忽略推荐后刷新页面 → 忽略记录丢失 → 系统重复推荐同样的内容。

**需要的表结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 主键 |
| canvasId | string | 哪个白板 |
| nodeIdA | string | 节点对 A（排序后） |
| nodeIdB | string | 节点对 B（排序后） |
| dismissedAt | string | 忽略时间 |

源码：`frontend/src/services/dexie-db.ts`（当前 v1-v5，需升级到 v6）

  │ 💡 系统发现 2 组概念可能有关联：                 │

**这就是用户现在完全看不到的东西。**

### 每条断裂管道让用户"少了什么"

#### Pipe#3 ❌ 没人调用 API → 整个推荐功能完全不存在

类比：外卖 App 的"下单"按钮坏了 — 餐厅菜做好了，骑手也在等，但你点不了餐。

`fetchRecommendations()` 写好了但没有代码去调用它。后端再怎么智能，前端永远不会去问"有什么推荐？"

→ **你的体验**：白板上放再多节点，永远不会看到任何推荐。

#### Pipe#4 ❌ 没有推荐界面 → 就算拿到数据也没地方展示

类比：快递到了楼下，但家里没信箱也没门铃。

没有 RecommendationBar 组件，没有卡片展示推荐内容，没有"接受/忽略"按钮。

→ **你的体验**：屏幕上没有任何地方展示推荐结果。

#### Pipe#5 ❌ 推荐边没有样式 → 分不清哪些是系统推荐的

类比：助手帮你贴了便签，但颜色和你自己贴的一模一样，过一会分不清谁贴的。

`isRecommended` 硬编码为 false → 所有连线都是实线，无法区分来源。

→ **你的体验**：接受推荐后，新连线和手动画的线完全一样，无法辨别。

#### Pipe#6 ❌ 缺忽略记录表 → 忽略了刷新就丢

类比：你跟推销员说"别再打了"，但他没记下来，第二天又打过来。

`dismissed_recommendations` 表不存在 → 刷新后忽略记录消失。

→ **你的体验**：每次打开白板，同样的推荐反复出现，忽略了也没用。

#### Pipe#7 ❌ 忽略不能告诉后端 → 后端不知道你拒绝过什么

和 #6 是一对。本地没地方存，也没法发给后端。后端每次都传空数组，不知道你忽略过什么。

→ **你的体验**：和 #6 合力造成"忽略功能完全不工作"。

#### Pipe#8 ❌ 没有触发机制 → 系统不知道什么时候该分析

类比：请了学习助手但没告诉他什么时候该给建议，他就一直站着什么都不做。

规范要求节点数 ≥ 5 + 操作后等 5 秒才触发。实际完全没有这个逻辑。

→ **你的体验**：推荐永远不会自动出现。

#### Pipe#9 ⚠️ 标签建议被丢弃 → 新连线没有自动标签

后端分析了你最常用的标签（如"前置知识"），但前端 `addEdge()` 传空字符串。

→ **你的体验**：接受推荐后连线没有标签文字，需要手动补。

#### Pipe#10 ❌ 忽略记录不跨设备 → 换设备又推荐一遍

Sync Outbox 只认识 node/edge/board，不认识 recommendation。

→ **你的体验**：笔记本上忽略的推荐，换台式机又出现。

### 功能影响汇总

| 用户功能 | 被哪些管道阻断 | 影响程度 |
|---------|---------------|---------|
| ❶ 看到推荐 — "系统建议我连这两个概念" | Pipe#3 + #4 + #8 | 100% 不可用 |
| ❷ 接受推荐 — "点接受，自动画虚线连接" | Pipe#4 + #5 + #9 | 100% 不可用 |
| ❸ 忽略推荐 — "点忽略，以后别再推这对" | Pipe#4 + #6 + #7 | 100% 不可用 |
| ❹ 多设备一致 — "笔记本忽略的台式机也忽略" | Pipe#10 | 增强功能 |

### 最小可用路径（MVP 思维）

```

虽然：没虚线样式、标签为空、刷新丢忽略记录
但是：核心功能已经可用

先打通关键路径，再补完细节。
```
