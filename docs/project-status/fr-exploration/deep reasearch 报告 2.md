## T1：6路RAG融合算法可信度审查

### RRF 算法本身是成熟的，但"6路"是问题所在

**Reciprocal Rank Fusion（RRF）** 由 Cormack、Clarke 和 Büttcher 在 SIGIR 2009 发表，已积累超过344篇学术引用。算法公式为 `RRF_score(d) = Σ 1/(k + rank_r(d))`，在所有 TREC 测试实验中都超越了 Condorcet Fuse 和 CombMNZ，并在几乎所有情况下超越了单一最优排名。 RRF 不需要训练、无需归一化、天然适合融合分数不可比较的异构来源——这正是你们用它来融合知识图谱遍历分数和向量余弦相似度的原因。

RRF 目前已是 Elasticsearch、Azure AI Search、LangChain、Weaviate、Pinecone 的默认融合方法，**本身没有问题**。

**真正的问题是：6路来源太多。** 文献记录了明确的失效模式：

FAIR-RAG 的研究表明，第四次检索迭代"在所有复杂数据集上持续导致平均答案质量下降"。更关键的是，SIGIR 2024 的研究发现，将随机文档加入上下文使 LLM 准确率提升了 35%，而高置信度但不相关的检索结果实际上会**降低**准确率——这意味着弱来源给出的"自信但错误"的结果比随机噪音更有害。

2025 年的工程实践共识已经形成：多路检索 → RRF 融合（取 top 50-100） → **cross-encoder 重排序** → 最终取 top 5-10 喂给 LLM。 你们目前有质量检查层，但缺少专门的 cross-encoder 重排序阶段——这是单一改进效果最大的地方，重排序相比纯向量检索可带来 15-40% 的精度提升。

**知识图谱 + 向量检索的混合方案在学术上已有充分验证。** HybridRAG（Sarmah 等人，ICAIF 2024，NVIDIA/BlackRock）表明 KG + 向量检索在忠实度（0.96）、答案相关性（0.96）和上下文召回率三项指标上均优于任何单一方法。 你们 Graphiti + LanceDB 的组合方向是对的。

**具体建议**：将 6 路精简到 3-4 路（KG + 向量搜索 + 多模态 + Vault Notes，如果后两者不重叠），在 RRF 之后加入 cross-encoder 重排序（推荐 Cohere Rerank v4.0 或 bge-reranker），并对每个来源设置置信度门槛，低于门槛的结果在进入融合前就过滤掉。

## T2：记忆注入触发时机对抗性审查

### 现有后端方案：fire-and-forget 有三个致命缺陷

当前实现是：Agent 回答 → `asyncio.create_task()` → `episode_worker.enqueue()` → `graphiti.add_episode()`。这个模式在生产环境下有三处已记录的失效点：

**缺陷 1：Python 3.12+ 的垃圾回收会静默终止任务。** `asyncio.create_task()` 只创建对 task 的弱引用。如果 task 没有被强引用持有，它可能在执行过程中被垃圾回收器静默终止，没有任何错误提示。 Python 官方文档现在已明确对此发出警告。

**缺陷 2：异常被静默吞掉。** 未被 await 的任务中，异常被封装在 Task 对象内部，永远不会浮出到监控系统。如果 `graphiti.add_episode()` 报错，没有人会知道。

**缺陷 3：服务器崩溃 = 全部数据丢失。** FastAPI 内置的 BackgroundTasks 在同一进程中按顺序执行，没有持久化、没有监控、没有可观测性。FastAPI 官方文档本身就建议将 Celery 用于任何需要持久性的操作。 你们现有的 `dead_letter_episodes.jsonl` 只能捕获已处理的失败，无法处理崩溃。

**对比 Claude Code Hook 方案：** Claude Code 的 PostToolUse/Stop hook 是**同步执行**的——它作为 Agent 控制流的一部分确定性地运行，失败时可以阻断后续动作，有明确的退出码错误处理机制。这在可靠性上天然优于 FastAPI 的后台任务。代价是延迟：每次工具调用都要增加处理时间。

**推荐修复：SQLite 持久化队列（约 30 行代码）**

用 `persist-queue` 库的 `SQLiteAckQueue` 替换 `asyncio.create_task()`：

```python
# 替换前（不可靠）
asyncio.create_task(episode_worker.enqueue(episode_data))

# 替换后（崩溃安全）
from persist_queue import SQLiteAckQueue
episode_queue = SQLiteAckQueue('episodes.db', auto_commit=False)
episode_queue.put(json.dumps(episode_data))  # 原子写入，崩溃不丢失
```

后台线程从队列消费，成功后 ack，失败时自动重试，服务重启后继续处理未完成的项目。这把现有"失败后补救"的 dead letter 模式，升级为"先持久化再处理"的根本性可靠性保证。

## T3：L1路由器 — 两条路对比探索

### 现有 LangGraph 路由方案的问题

在多路 RAG 系统中，路由分为四种标准方式：规则路由（约0ms，快但脆弱）、语义路由（约50-100ms，基于嵌入相似度）、微调分类器（约10ms，高精度但需标注数据）、LLM 路由（约200-500ms，最灵活）。

研究强烈支持**选择性激活**而非全量检索：RouteRAG 的研究表明，RL 驱动的自适应路由——对多跳问题选择图检索、对简单事实选择文本检索——比静态基线高出 6-7 个 F1 点。 每次查询激活 1-3 路，比激活所有 6 路效果更好，原因是减少了噪音注入和 tokens 消耗。

### 两条路的对比分析

**路径 A：现有 LangGraph 规则路由**

优点是零延迟、完全可调试。缺点是对你的使用场景致命——你的查询用中文、有个人简写、往往是探索性的，规则几乎无法覆盖这些情况，会频繁路由失败。

**路径 B：Claude Sonnet 作为 Smart Agent Router**

三篇顶级论文支持这个方向。Self-RAG（Asai 等人，ICLR 2024）训练 LLM 发出反射 token 来控制按需检索，在开放域问答上超越 ChatGPT。 FLARE（Jiang 等人，EMNLP 2023）使用前瞻性主动检索，仅在模型检测到低置信度 token 时才触发检索。 两者都指向同一结论：不是所有查询都需要相同的检索策略。

**对于你的单用户个人学习系统，成本计算明确支持 LLM 路由。** 每天约 50-200 次查询，每次路由调用约 $0.005，日均成本不足 $1。200-500ms 的开销在 1-5 秒的总检索+生成周期中只是很小的比例。而你的查询语言多变、有个人化语境、经常是探索性的——这恰好是 LLM 路由擅长、规则路由失败的场景。

**推荐架构：快速规则 + LLM 兜底的混合方案**

```python
def route_query(query: str) -> List[Source]:
    # 快速路径：关键词明确的查询直接路由（0ms）
    if "我的笔记" in query or "vault" in query.lower():
        return [Source.VAULT_NOTES]
    if "图片" in query or "截图" in query:
        return [Source.MULTIMODAL]
    if "有什么关系" in query or "联系" in query:
        return [Source.GRAPHITI_KG]
    
    # 智能路径：模糊查询交给 Claude Haiku（~150ms）
    return llm_router.decide(query)  # 输出 1-3 个来源
```

## T4：FSRS + Graphiti 融合调用链对抗性审查

### 正确顺序是"图谱优先门控，FSRS 次要排序"

文献和所有生产级学习系统都遵循同一架构：**知识图谱决定候选资格，间隔重复在合格集内决定优先级**。这是你的方案 A。

Math Academy 的 FIRe 模型首先识别学生"准备好学习的"主题（所有前置已掌握），然后用 SR 调度确定这些就绪主题中哪些应该复习或学习。 ALEKS 基于知识空间理论，始终引导学生学习"当前恰好准备好的内容"。Khan Academy 要求前置技能达标后才解锁后续技能。

**这不只是设计偏好，而是有直接实验数据支撑。** Baker 等人研究了 7913 名学生的 250783 次任务，发现：前置已满足的任务在数学上被掌握的可能性高出 **71%**（p < 0.001），在英语上高出 **7.7倍**（p < 0.001）。

### 现有 0.4/0.3/0.3 公式的致命 Bug

```
priority = 0.4*(1-p_mastery) + 0.3*(1-retrievability) + 0.3*kg_relevance
```

这个公式把前置依赖关系当成可以用其他因子"补偿"的权重，导致直接的逻辑错误：一个前置概念完全未掌握、但 kg_relevance=1.0、retrievability=0 的概念，优先级得分 = **1.0（满分）**，系统会优先考察它——而这几乎必然让学生失败。

**正确架构是两阶段：**

```python
# 第一阶段：硬约束门控（脚本，不碰 AI）
eligible = [c for c in all_concepts
            if all_prerequisites_met(c, threshold=0.7)]

# 第二阶段：在合格集内用优先级排序（这里的权重才有意义）
for concept in eligible:
    priority = 0.5 * (1 - mastery) + 0.3 * (1 - retrievability) + 0.2 * recency_penalty
```

把 `kg_relevance` 从加权项改成门控条件，是这个公式最重要的一次修改。

### FSRS 存在 Neo4j 节点属性上是否合理？

对于单用户系统，这个存储选择**可以接受**，但有一个必须注意的细节：FSRS 状态必须是"每用户每概念"的，应该存在 `(:User)-[:LEARNING_STATE]->(:Concept)` 中间关系上，而不是直接挂在共享的概念节点属性上。否则多用户场景下数据会互相覆盖（即便你现在是单用户，这是未来扩展的隐患）。

## 考察方案设计
  
### 方案一：交错练习（推荐优先实施，证据最强）

Plan-and-Execute 本质上是**集中练习**（blocked practice）——几十年认知科学研究证明这是效果最差的记忆策略。交错练习（把不同主题混合在一次学习中）的实验数据令人震惊：Taylor & Rohrer（2010）发现交错练习在延迟测试中得分 77% vs 集中练习 38%，效应量 d = 1.21；Rohrer 等人（2015）发现交错练习使测试得分提高了三倍，d = 1.34。这是教育干预研究中最大的效应量之一。

与你现有系统的整合非常简单：FSRS 已经提供了调度引擎，你只需要在选题时**主动交错不同 group_id 的概念**，而不是按最近学习的主题集中出题。在优先级公式里加一个"交错加分项"：当前题目的主题与上一道题不同时，给它的优先级加权。**实现周期约 1 周，效果却可能是所有方案里最大的。**

### 方案二：苏格拉底式对话考察

AutoTutor（Graesser 等人）在 10 余次对照实验中实现了平均效应量 d = 0.8，物理学科效果达到 d = 0.75-1.22。关键发现：苏格拉底式对话主要提升**深度学习**（推断、解释），而非浅层记忆——对 CS 和数学这类需要概念理解的学科尤其适合。

在你的系统里，Graphiti 存储每个概念的"期望理解点"和"常见误解"作为对话脚手架，LanceDB 检索的原文片段用于为 LLM 的苏格拉底提问接地。FSRS 把对话质量评分作为复习信号更新卡片状态。**实现周期约 2-3 周。主要代价是每个概念的覆盖速度变慢，且对话得分较难精确映射到 FSRS 等级。**

### 方案三：概念图谱验证考察

不考察孤立概念，而是考察**概念之间的关系**——让学生解释为什么 A 和 B 有关联、识别错误的连线、预测图谱结构变化的后果。Ruiz-Primo & Shavelson 发现概念图打分的评分者间信度 > 0.90，与 MCQ 测试的相关性 r = 0.31，说明概念图捕获了传统测试无法测量的结构性知识。

这个方案对你的系统有**独特的天然适配性**：Graphiti/Neo4j 本身就是概念图，测试直接检验学生心智模型是否与图谱结构匹配。FSRS 可以调度**关系边**的复习，而不只是单个概念节点。**实现周期约 2 周。**

### 方案四：错误模式挖掘 + 靶向纠偏

Carnegie Learning 的 Cognitive Tutor 基于 ACT-R 认知理论，用"错误规则"编码常见误解，已在 27 项独立研究中得到验证，被美国教育部列为五个典范课程之一。Corbett & Anderson（1991）证明误解靶向即时反馈将学习时间缩短了 **3 倍**。

整合方案：在 Graphiti schema 里加 `HAS_MISCONCEPTION` 关系，为 CS61A、CS61B、Math 53 各自建初始错误模式库，FSRS 为检测到的误解创建稳定性极低的专项复习卡。**实现周期约 3-4 周，需要持续维护错误模式库。**

### 方案五：计算机自适应测试（CAT/IRT）

CAT 使用项目反应理论（IRT）动态选择下一道题，使测量精度最大化，与固定测试相比能用 **50-80% 更少的题目**达到等效精度。这是 GRE、GMAT、NCLEX-RN 背后的数学基础。

对于你的单用户系统，主要挑战是**冷启动问题**：IRT 参数标定传统上需要大量被试数据。可行的做法是"轻量版 CAT"：FSRS 的 retrievability 已经是 P(correct | θ) 的近似，用它作为初始能力估计，用 LLM 生成题目并在线标定难度参数。**实现周期约 3-4 周，是五种方案中工程复杂度最高的。**
