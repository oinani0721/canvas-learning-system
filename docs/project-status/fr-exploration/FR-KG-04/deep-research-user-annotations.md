## DR1-1: group_id 三级考察模式

**来源**: Deep Research 报告 1, Line 90-92
**所在章节**: 方案 #2 — 跨白板关联改为手动 + group_id 方案

**批注原文**:
> 我的个人觉得，只要我们的白板用同一个学科用同一个group id 那么，那么这些白板就是同一个数据库，那么这个时候使用检验白板的考察就是对这个学科来进行考察了，那么再往下划分，那么就是单个白板的进行检验考察，然后再往下就是 单个节点的检验检测了
> 请问你觉得需要这样设置考察方式吗？这样的 3 模式考察真的会对学习有很大提升吗

**挂钩内容**: group_id 设计与考察粒度

**用户的想法**:
```
三级考察粒度：
  Level 1: 学科级 (group_id = "数学") → 考察整个数学学科
  Level 2: 白板级 (group_id = "数学:离散数学") → 考察单个白板
  Level 3: 节点级 (node_id) → 考察单个概念
```

**Deep Research 待回答**: 这种三级粒度在教育学上是否有依据？是否需要？

## DR3-0: 总体批评（最重要）

**来源**: Deep Research 报告 3, Line 1-5
**所在章节**: 文件开头（用户对前次分析的批评）

**批注原文**:
> 我的提问：
> T2，答非所问，没有回答当前的机制，以及是否实现了 hook 机制，不仔细看代码
> T3，答非所问，没有进行 Langgraph 设计的具体讲解，没有深度的了解 claude code 作为 smartagent 给 dumb tool 来做的判断效果好不好，你也没有告诉我选择的方案是从这两者怎么脱出的
> T5；方案 1 到 4 都是我有采纳考虑的教学方案，但是我们这里讨论的不是出题考察模式，我们问的是 agent 整合出题的资源设计，我们为什么要选择 plan 的原因难道不是因为，保持 smart agent dumb tool 来整合资源出题的时候更加可靠，所以才选择了 plan，单纯的硬编码太蠢了，单独的 agent 太野了，所以我才要你思考有没有除了 plan 更加靠谱的方案。
> 请你重新 deep explore 调研。如果又有什么疑问，请你向我增量提问。

**核心关切（3 条）**:

1. **T2 记忆机制**: 需要回答"当前代码是否实现了 hook 机制来触发记忆注入/调出"，而不是泛泛讨论
2. **T3 RAG 路由**: 需要具体讲解 LangGraph 设计，对比 "smart agent 判断" vs "dumb tool 脚本" 哪个效果更好，并说明选择方案是如何从这两个极端中脱出的
3. **T5 出题资源设计**: 用户问的不是"出什么题"，而是"agent 如何整合资源来出题"。Plan 模式的价值在于：硬编码太蠢、纯 agent 太野，Plan 是中间态。需要探索 Plan 之外的方案

**⚠️ 这条批评直接影响下一轮 Deep Research 的方向和质量标准**
