## T2：内存队列的对抗性审查

### 实际代码情况

代码里不是简单的 `asyncio.create_task()` fire-and-forget，而是一个设计相当完整的 `GraphitiEpisodeWorker`：

- `asyncio.Queue(maxsize=100)` 内存队列
- 独立的 `_run()` 协程循环，串行处理
- 指数退避重试逻辑，每个任务有 `max_retries`
- `DeadLetterStore`（jsonl 文件）处理永久失败的 episode
- 30秒优雅停机排空超时

**Claude Code hook 机制在这个项目里不存在**（`PreToolUse / PostToolUse / Stop hook` grep 全部为空）。你在 Claude Code 里配置的 hook（`mutation_gate.sh`、`post-tool-router.sh` 等）是你的开发工作流层面的，和这个系统的 FastAPI 内存注入是完全独立的两套体系。

### 对抗性裁定：内存队列对单用户工具够用

单用户个人工具的实际崩溃丢失窗口极小。以对话交互频率来看，队列里在任意时刻的深度通常是 **0-3 个 item**。Python/FastAPI 的未处理异常会返回 500 错误但不会崩溃进程；asyncio 协程隔离保证单个 task 失败不影响整体。真正的进程崩溃（OOM kill、segfault）对正常运行的 Python Web 应用极少发生。开发期间最常见的"崩溃"是主动重启，已经被 30 秒优雅排空处理了。

Dead Letter Store（jsonl）已经覆盖了最常见的失败模式——LLM API 错误、Neo4j 超时、提取失败。这个架构和 Claude-Mem（一个基于 Claude Code hook 的记忆系统）独立选择的方案是一样的：hook 捕获 → 队列缓冲 → 后台 worker → Graphiti 处理。

## T3：L1 路由器对抗性审查

### 实际代码情况

L1 路由器是**纯关键词规则**，不是 LLM：

```python
def classify_query_intent(query: str) -> str:
    # file_keywords 匹配 → "file_locate"（3路检索）
    # history_keywords 匹配 → "learning_history"（3路检索）
    # 其他全部 → "comprehensive"（全 5 路）
    # 代码注释："MVP 阶段使用规则分类，LLM 分类为远期增强"
```

路由逻辑本身是关键词匹配，延迟 < 1ms。LangGraph 的 `add_conditional_edges` + `Send` 对象实现了并行 fan-out——这个基础设施本身是对的。但问题是：**绝大多数查询都落进 "comprehensive"（全 5 路）**，路由形同虚设。

### 对抗性裁定："comprehensive" 默认值有害

这不只是浪费计算资源。ICLR 2025 的研究（Long-Context LLMs Meet RAG）证明，**不相关的检索片段会主动降低 LLM 生成质量**，即使相关文档也同时存在。Shi 等人（ICML 2023）证实：LLM 会系统性地被不相关上下文误导。当你的路由器把大多数查询送到全 5 路时，它注入了与有效信号竞争的噪音。

Adaptive-RAG（Jeong 等人，NAACL 2024）发现：根据复杂度把查询路由到合适的检索策略，消除了把简单查询当复杂查询处理的开销——这正是你的 "comprehensive" 默认值在制造的问题。

### 推荐方案：三级混合路由

学术文献和生产系统都收敛到同一架构：越来越贵的分类器级联，便宜的方法处理简单情况，贵的方法处理模糊情况。

**第一级 — 关键词规则（< 1ms）**：保留现有关键词匹配，但改为**只在高置信度时生效**。只匹配明确无歧义的短语（"文件在哪"、"我的学习进度"），不明确时返回 `None` 而不是 "comprehensive"。这一级处理约 20-30% 的查询。

**第二级 — 语义路由（约 50ms）**：用 Aurelio Labs 的 `semantic-router` 库 + `FastEmbedEncoder`（本地运行，无 API 调用）。为每条路由定义 10-20 个示例话语，基于嵌入相似度分类。可以处理关键词匹配覆盖不到的同义表达和隐含意图。置信度低于阈值时返回 `None`，干净地升级到第三级。这一级处理约 40-50% 的查询。

**第三级 — Claude Sonnet（约 300-500ms）**：对剩余约 20-30% 真正模糊的查询做 LLM 分类。对单用户系统，300-500ms 的开销在 2-10 秒总流水线里可以忽略不计。如果和其他准备工作（构建系统提示、预热缓存）**并行运行**，在大多数情况下感知延迟为零。

关键路由改变：引入 **"概念解释" 路由**（1-2 路检索）来覆盖最常见的查询类型，让 "comprehensive"（全 5 路）成为例外而不是默认。

## T5：五种编排模式的对抗性分析

你描述的核心张力（脚本太死 / 纯 Agent 太野 / Plan-and-Execute 居中）隐含了一个假设：这个问题需要"规划"。当数据来源固定已知时，Plan-and-Execute 的规划步骤每次都生成同样的计划。**真正的问题不是"拿什么数据"，而是"怎么拿得好"以及"拿到的够不够"。** 这个重新定位决定了哪些模式真正有价值。

### 模式一：ReWOO — 对固定来源无益

**核心机制**：一次 LLM 调用规划所有工具调用，串行执行，最后一次 LLM 调用综合。总计恰好 2 次 LLM 调用，不管工具数量多少。

**和 Plan-and-Execute 的对比**：ReWOO 的效率优势来自比 ReAct（N+1 次调用）更少的 LLM 调用。但对这个系统它是错误的选择：规划器每次都会生成相同的计划（"从所有 5 个来源拿数据"），白白浪费一次 LLM 调用。更糟的是，ReWOO 的参考实现和 LangGraph 官方教程都**串行执行**工具调用——而你现有的 `Send` 对象已经在并行运行所有 5 路检索。ReWOO 实际上会把当前并行的操作变成串行，**增加延迟**。

**裁定：比 Plan-and-Execute 更差，不是更好。**

### 模式二：LLMCompiler — 解决的不是你的问题

**核心机制**：为每次查询动态生成工具调用的 DAG（含显式依赖边），流式调度引擎并行派发独立任务，Joiner 可以触发重新规划。比 ReAct 实现 **3.6 倍延迟加速**。

**和 LangGraph Send API 的对比**：LLMCompiler 动态生成 DAG；LangGraph 的 `Send` 对象在设计时静态定义。对 5 个固定独立来源，**LangGraph Send API 已经实现了 LLMCompiler 规划器会生成的结果**，只是没有 LLM 开销。LLMCompiler 的价值主张（动态 DAG 生成）只在工具图随查询变化时才体现。LangChain 团队自己的注释也指出："规划器输出解析格式脆弱"，"变量替换脆弱"。

**裁定：和 Plan-and-Execute 的同类问题，不适合固定来源场景。**

### 模式三：Workflow 槽位填充 — 你现在就应该做的

**核心机制**：预定义工作流 DAG 决定做什么（固定结构），LLM 只负责填写参数（智能槽位）。这正是 Anthropic 官方"构建有效 Agent"指南中的**并行化工作流模式**。

**和你现有系统的关系**：**你现有的 fan-out/fan-in 就是"固定结构"的一半。** 缺少的是"智能槽位"——一个在 fan-out 之前运行的 LLM 参数化节点，基于学生的掌握度状态为每个数据来源生成优化的查询参数。

举例：现在向 LanceDB 发的查询是静态的 "concept: mitosis"，换成参数化节点后可以根据 FSRS 数据生成 "mitosis cell division 和 meiosis 之间的常见混淆点"——因为系统知道这个学生在区分二者上有困难。向 Neo4j 查前置依赖时，高掌握度概念用 1-hop，低掌握度概念用 3-hop。向 Graphiti 查时，根据历史错误模式过滤出对出题最相关的误解记录。

**实现成本**：在现有 `Send` fan-out 之前加一个 LLM 节点，结构化输出 5 个来源各自的查询配置。约 **1-2 天**工作量，无需改动现有图结构。

**裁定：这是最适合你场景的模式，而且你已经有 80% 的基础设施了。**

### 模式四：DSPy 声明式程序 — 优化层，不是编排层

**核心机制**：定义程序签名（输入: `concept_id, mastery_state` → 输出: `exam_question with rationale`），DSPy 自动优化 prompt 链，包括决定调用哪些检索器。

**对这个系统**：DSPy **不做编排**。多来源检索协调在你的 `forward()` 方法里是普通 Python，DSPy 没有原生的多检索器支持（GitHub issue #143，至今未关闭）。DSPy 最好理解为可以放在 LangGraph 节点内部的 **prompt 优化层**，不是 LangGraph 的替代品。

此外，**API 稳定性是实际风险**：2025 年 8 月从 v2.6 到 v3.0 有重大 breaking change，多个 PyPI 版本被撤回，有用户报告优化器运行花了 $2 LLM 费用后崩溃。

**裁定：等编排架构稳定后再作为可选优化层引入，不适合作为主要架构选择。**

### 模式五：后验验证（Critic-in-the-Loop）— ROI 最高的改动

**核心机制**：检索完成后，用一次轻量 LLM 调用评估收集到的上下文是否充分，不足时触发针对性补充检索。这就是 CRAG（Yan 等人，2024）的模式应用到多来源场景。

CRAG 论文中，用 T5-large（0.77B 参数）做专注评估器，在 PopQA 上提升精度 **19%**，在传记生成上提升 **14.9%**。关键发现：专注的小型评估 prompt 效果优于 ChatGPT——任务的针对性比模型能力更重要。

**为什么它解决了其他模式都解决不了的问题**：向量数据库可能返回语义不匹配的结果；FSRS 数据可能过期；前置依赖图对这个特定概念可能不完整。这些是**静默检索失败**，无论多少规划都无法在执行前预防——只能在检索后检测。

**对这个系统的具体实现**：fan-in 之后加一个 LLM critic 节点，接收所有 5 路检索结果，针对具体标准评估完整性："前置概念信息是否存在？FSRS 数据是否有效？向量结果是否语义对齐？图谱覆盖是否充分？"任何检查失败时，条件边路由到**针对性单路补充检索**（最多重试 1 次）。每题增加约 **$0.0001-0.0005** 费用——实际可以忽略。
