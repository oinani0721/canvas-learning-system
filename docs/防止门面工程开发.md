# 生产团队如何真正防止 AI 门面工程

**简短结论：没有任何单一方法能解决这个问题，但将评测驱动开发、确定性强制执行 Hooks、真实服务集成测试和结构化上下文注入组合成分层策略，可以将返工率降低 70% 以上。** 业界已将"评测驱动开发"（EDD）确立为 AI/LLM 系统的 TDD 等价方法——用带阈值的量化评测替代二元通过/失败的单元测试，用评测数据集替代规格文档。这一方法已在 Stripe、Notion、Vercel、Shopify 和 Nubank 得到验证。针对防止门面工程，最有效的模式是将上下文隔离子 Agent（测试编写者和实现者使用独立上下文窗口）、阻断无失败测试时不得实现代码的确定性 Hooks，以及对真实服务进行合同优先验证相结合。BMAD V6 的失败模式——验证文档完整性而非运行时行为——是一个所有成熟团队都已摒弃的反模式。

## 一、评测驱动开发已取代 AI 系统中的 TDD

经典 TDD 假设输出是确定性的：给定输入 X，期待输出 Y。基于 LLM 的系统从根本上打破了这一假设。学术论文《LLM Agent 的评测驱动开发与运营》（EDDOps，Xia 等，CSIRO/UNSW，arXiv:2411.13768）分析了 **134 篇学术论文和 27 个行业案例**，得出结论：由于 Agent 具有非确定性、开放性，且随时间推移受系统级交互影响，"TDD 和 BDD 等经典软件工程方法面临局限"。业界的答案是 EDD：在构建之前，通过量化评测定义"优质"的含义，然后迭代直到评测通过。

最具生产验证的实现工具：

### promptfoo（~25,600 GitHub stars，2026 年 3 月被 OpenAI 收购）

这是最明确支持测试优先的框架。其文档指出："目标：测试驱动的提示词工程，而非试错法。" 工作流与 TDD 完全一致——先建立定义期望行为的测试用例，运行 `promptfoo eval` 对提示词进行评测，迭代直到通过。支持丰富的断言类型（`equals`、`contains`、`is-json`、`llm-rubric`、自定义 Python），当质量分数低于阈值时触发 CI/CD 阻断，并提供专用 GitHub Action。**127 家财富 500 强企业**在使用它。

### DSPy（斯坦福 NLP，~25,000 stars）

这是结构上与测试优先思维最契合的框架。其整体理念从定义度量函数开始——等同于先写测试——然后让优化器（MIPROv2、BootstrapFewShot）自动搜索使这些度量最大化的提示词。`dspy.Assert`（违反时停止流水线的硬约束）和 `dspy.Suggest`（软性引导）本质上是嵌入程序中的运行时测试断言。学术研究表明，断言将规则合规性提升了最高 **164%**，质量提升了最高 **37%**。

### DeepEval（~13,900 stars，Apache 2.0）

将 pytest 范式直接引入 LLM 测试。你编写包含输入、期望输出和上下文的 `LLMTestCase` 对象，然后使用 50+ 个基于研究的指标定义质量阈值。`deepeval test run` 命令原生集成到 CI/CD。对于包含评分组件的 AI 学习应用，DeepEval 的忠实度、幻觉和上下文精确率指标会直接捕获以字符长度启发式方法伪装成真实 AI 评分的情况——因为这些指标评估的是语义质量，而非表层形式。

### Braintrust（2026 年 2 月完成 8000 万美元融资，估值 8 亿美元）

提供了 EDD 在规模上有效的生产证据。Notion 工程团队在使用 Braintrust 的评测门控实施系统性评测后，从**每天修复 3 个问题提升到 30 个**。Stripe 针对 11 个不同环境构建了全面基准测试，使用确定性评分器通过真实 API 调用和自动化 UI 测试验证软件——验证真实 Stripe API 对象存在于数据库中，而不仅仅是代码能编译通过。

## 二、TDAD 论文证明：定向测试上下文可将回归降低 70%

最直接相关的学术发现来自《TDAD：测试驱动的 Agentic 开发》（arXiv:2603.17973），该研究通过基于图的测试影响分析，在 SWE-bench Verified 上实现了 **70% 的回归降低**（从 6.08% 降至 1.82%）。

论文最令人震惊的发现：在没有定向测试上下文的情况下添加通用 TDD 程序性指令，实际上将回归增加到 9.94%——比什么都不做还要糟糕。这一洞见意义深远：**Agent 不需要被告知如何执行 TDD；它们需要被告知检查哪些测试。** 这直接解释了 BMAD V6 失败的原因——它提供的是流程文档，而非有针对性的、可执行的测试上下文。

互补论文《TGen：代码生成的测试驱动开发》（arXiv:2402.13521）证实，测试信息显著提升了代码生成准确性，尤其是在复杂任务上。当测试既作为规格又作为验证工具，并通过提供反馈的修复循环时，AI Agent 能产出质量大幅提升的实现。

### LangWatch Scenario（~210 stars，快速增长中）

这是最贴近实践的真正测试优先 AI Agent 开发工具。它实现了"AI Agent 的领域驱动 TDD"——先编写场景测试（使用自然语言定义成功/失败标准），然后实现 Agent 使其通过。该框架使用 Agent 对 Agent 的测试方式：判断 Agent 和用户模拟器根据业务规则评估你的 Agent。

## 三、真正防止门面工程的五层防御

没有任何单一技术能防止门面工程。最有效的方法是叠加互补的防御层，每层捕获不同类别的门面。

### 第一层：用于测试编写的上下文隔离子 Agent

最具结构完整性的防御来自 alexop.dev 为 Claude Code 记录的"子 Agent 隔离模式"。测试编写者和实现者 Agent 运行在**独立的上下文窗口**中——测试编写者对计划中的实现一无所知，实现者只能看到失败的测试。这消除了"上下文污染"问题，即 LLM 下意识地围绕已经计划好的实现来设计测试。

配合 **TDD Guard**（github.com/nizos/tdd-guard，1,128+ forks），它阻止 Claude Code 在没有失败测试的情况下编写实现代码，从而创建 CLAUDE.md 规则单独无法提供的结构性强制执行。

### 第二层：确定性 Hooks 优先于建议性规则

Anthropic 自己的文档指出，CLAUDE.md 规则是"由可以被说服的 LLM 在运行时解释的"。确定性 Hooks 无法被绕过。关键 Hook 模式：

- **PostToolUse Hooks**：每次文件编辑后自动运行测试（退出码 2 阻断操作）
- **PreToolUse Hooks**：在 `create_pull_request` 上要求所有测试通过后才能创建 PR
- **SessionStart Hooks**：在上下文压缩后重新注入 TDD 规则

配套这些 Hooks 的实用 CLAUDE.md 反门面规则：
- 没有失败的测试，不得编写实现
- 提交实现前先提交失败测试
- 永远不要修改测试来让其通过——修复实现
- 优先选择集成测试而非模拟（Mock）
- 生产代码中不得出现 Mock 导入

### 第三层：使用 Schemathesis 进行合同优先强制执行

**Schemathesis**（~2,200 stars）读取 OpenAPI/GraphQL 规格，并使用 Hypothesis 自动生成数千个基于属性的测试用例。它发送真实 HTTP 请求并根据 Schema 验证响应。返回硬编码响应的门面实现会失败，因为 Schemathesis 生成的边界用例是门面无法处理的，并测试有状态工作流（POST → GET → DELETE 序列）。在 **Red Hat、Capital One 和 Mattermost** 生产环境中使用，学术评估显示它比竞争工具多检测 **1.4–4.5 倍**的缺陷。

### 第四层：通过 Testcontainers 进行真实服务集成测试

"被 Graphiti 库调用伪装的 Neo4j Cypher 查询"这一具体失败模式，可通过 Testcontainers 为测试启动真实 Neo4j Docker 实例来防止。当测试针对真实图数据库执行实际 Cypher 查询时，返回硬编码数据的模拟实现会立即失败。

Testcontainers 支持 50+ 个预构建服务模块，并与 GitHub Actions 和 GitLab CI 原生集成。成本/速度权衡通过以下方式管理：跨测试套件共享的单例容器、并行启动，以及 `RUN_INTEGRATION_TESTS` 环境变量。

### 第五层：针对生产环境中模拟代码的静态分析

**Semgrep 自定义规则**可以检测代码库中描述的确切反模式——`USE_MOCK_*` 标志、测试库在生产文件中的导入、伪装成真实实现的硬编码返回值。

**ArchUnit**（Java）和 **NetArchTest**（.NET）强制执行架构适应性功能：服务层必须依赖真实的 Repository 接口，而非 Mock 类。SonarQube 2025.1+ 包含 **AI 代码保证**功能，可自动检测 AI 生成的代码并应用更严格的质量门控。

## 四、真正适用于棕地代码库的 Claude Code 工作流

三种替代 BMAD V6 的方案在社区中获得了大量采用，每种都专为现有代码库而非全新项目设计：

### obra/Superpowers（~29,000 stars，2026 年 1 月进入 Anthropic 官方插件市场）

默认执行 RED/GREEN TDD——Agent 先写失败测试，只实现足以通过测试的代码，然后继续下一步。内置代码审查 Agent 根据计划、编码标准和架构原则评估实现。Simon Willison 称其创建者为"编码 Agent 最具创意的用户之一"。相比 BMAD 的关键优势：Superpowers 基于技能且轻量，而 BMAD 需要 12+ 个 Agent 角色和 34+ 个工作流。

### OpenSpec（Fission-AI，~28,000 stars）

明确"为棕地而非绿地设计"。使用 **Delta 标记**（ADDED/MODIFIED/REMOVED）相对于现有功能跟踪变更——正是这次提问所描述的"最小规格"或"Delta 规格"模式。工作流（`/opsx:propose` → `/opsx:apply` → `/opsx:archive`）只为变更的内容生成规格。代码生成后自动运行单元测试，只有在测试通过后才将任务标记为完成。它还可以从现有源代码反向生成规格，非常适合遗留代码库。

### GitHub Spec Kit（github/spec-kit）

遵循 Constitution → Specify → Plan → Tasks 工作流，任务分解明确包含"测试驱动开发结构——测试任务被包含并排序在实现之前编写"。然而社区反馈（GitHub Issue #1436）指出，通用模板不能反映现有代码库架构，限制了棕地场景的有效性。

对于当前特定技术栈（Tauri + React + FastAPI + Neo4j + LanceDB），**Tweag Agentic Coding Handbook**（tweag.github.io/agentic-coding-handbook）提供了一个与框架无关的 TDD 方法论，核心洞见是"测试即提示词"——测试成为引导 AI 走向精确预期行为的自然语言规格。

## 五、用于开发上下文的知识图谱：正在涌现但尚未成熟

关于知识图谱能否维护开发上下文（架构决策、已知陷阱、技术债）以防止重复犯错，答案比较微妙——工具存在，但采用仍处于早期阶段。

### Graphiti（Zep，~20,000 stars）

目前最有能力的选项。基于 Neo4j，具有时序事实管理（事实具有有效期窗口），在深度记忆检索基准上优于 MemGPT（**94.8% vs 93.4%**）。Zep 的 Daniel Chalef 演示了使用 Graphiti 的 MCP 服务器为 Cursor IDE 添加持久记忆，使用自定义实体类型（Requirement、Preference、Procedure）存储编码标准和项目规格。Agent 在采取行动前查询 Graphiti，检索存储的偏好和架构约束。这是使用知识图谱进行开发工作流记忆的最接近生产的实现——但需要 Neo4j 基础设施，增加了相当大的复杂性。

### mcp-neuralmemory（github.com/Hexecu/mcp-neuralmemory）

专为 AI 编码 Agent 设计。在 Neo4j 中追踪：目标与状态、约束与规则（架构决策、禁止模式）、策略与结果（有效/失败的内容，含自动学习），以及代码关系。策略与结果追踪直接解决了防止重复犯错的问题（如门面工程）——系统从失败中学习并将该上下文注入未来的 Session。

### 实用替代方案：AGENTS.md + 结构化 Memory 文件

对于大多数团队，目前更实用的方法更简单。**AGENTS.md** 已成为 AI 编码上下文的跨工具标准，目前有 25+ 个工具支持（Codex、Copilot、Cursor、Windsurf、Claude Code、Devin），GitHub 上已有 **60,000+ 个仓库**采用。由 Linux 基金会的 Agentic AI Foundation 支持，AWS、Anthropic、Google、Microsoft 和 OpenAI 均背书，正在成为"AI 辅助开发的 .gitignore"。

配合 CLAUDE.md 的工具特定功能，以及使用"文件即真相"模式的结构化 `memory/` 目录（决策、状态变更和理由存储在人类可读的 Markdown + JSONL 中，通过 git 版本控制），可在无需图数据库开销的情况下实现有效的错误预防。

### Code-Graph-RAG（代码结构专用）

**Code-Graph-RAG**（github.com/vitali87/code-graph-rag）使用 Tree-sitter 将多语言代码库解析为知识图谱，存储在 Memgraph 中，并通过 MCP 服务器集成到 Claude Code。arXiv:2601.08773 的研究发现，纯向量 RAG 在多跳架构推理上（控制器 → Repository 链、接口驱动的连接）"频繁失败"，而基于 AST 的知识图谱提供确定性的、编译器可见的依赖关系，防止 AI 生成结构性断裂的门面代码。

## 六、针对 Tauri+React+FastAPI+Neo4j+LanceDB 技术栈的具体建议

基于所有研究发现，以下是消除该特定代码库中门面工程的优先级实施计划：

### 第一周（立即执行）

- 安装 TDD Guard，配置 PostToolUse Hooks，每次文件编辑后运行 `pytest`
- 添加反门面 CLAUDE.md 规则块
- 用 OpenSpec 替换 BMAD V6，用于棕地 Delta 规格
- 添加 Semgrep 规则，检测 `USE_MOCK_*` 标志和生产代码中的测试导入

### 第 2–4 周（短期）

- 在 CI 中使用 Testcontainers 启动真实 Neo4j 和 LanceDB 实例
- 编写验证真实 Graphiti 库调用的集成测试（而非伪装成 Graphiti 的 Cypher）
- 为所有 AI 评分组件实现 promptfoo 或 DeepEval 评测套件——这将立即捕获字符长度启发式方法，因为它们评估的是语义质量
- 针对 FastAPI OpenAPI 规格配置 Schemathesis

### 第 2–3 个月（中期）

- 实现子 Agent 隔离模式——测试编写者和实现者使用独立上下文窗口
- 在 CI 中添加变异测试（Python 用 mutmut，TypeScript 用 Stryker），设置最低变异分数阈值
- 部署 Graphiti MCP 或 mcp-neuralmemory，维护架构决策和已知陷阱的知识图谱

### 持续进行

- 运行基于属性的测试（Python 用 Hypothesis，TypeScript 用 fast-check），定义门面实现无法满足的不变量
- 将 AGENTS.md 作为跨工具的上下文标准维护
- 在结构化记忆文件中追踪策略与结果，让系统从每次门面工程事故中学习

## 结语

研究揭示，门面工程并非新问题——它是一个有充分文献记录的失败模式，当规格验证形式而非功能时就会出现。TDAD 论文中 **70% 的回归降低**和 Notion 的 **10 倍吞吐量提升**证明了评测驱动、测试优先的方法在规模上确实有效。

核心洞见是：CLAUDE.md 规则和规格文档都是建议性的——LLM 可以被"说服"忽略它们。只有确定性强制执行（阻断操作的 Hooks、拒绝 PR 的 CI 门控、针对真实服务进行测试的合同验证器）才能提供可靠的防护。

最有效的反门面策略是结构性的：将测试编写和实现隔离在独立上下文中，针对真实服务运行测试，并使用变异测试验证测试质量。没有任何单一工具能解决这个问题，但本文描述的五层防御体系使 AI 极难生成看起来合理但实际上是假的实现并通过完整流水线。
