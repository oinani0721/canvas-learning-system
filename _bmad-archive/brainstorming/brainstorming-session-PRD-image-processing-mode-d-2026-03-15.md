# PRD 图片处理方案更新 — Mode D 架构下的简化

> 来源 Session：PRD-Review 讨论 Tab（2026-03-15）
> 触发：用户提问"如果把对话框架换成 Claude Code/OpenCode，图片是不是可以直接处理？"
> 状态：决策已确认，Decision-Review PENDING

---

## 一、讨论背景

PRD 当前描述图片处理为"V1 伪多模态方案"——即所有图片都需要先通过 Vision API 做 OCR 提取文字，再将文字注入 AI 对话上下文。

用户发现：既然 Mode D 架构已确认使用 Claude Code/OpenCode 作为对话骨架，而这些工具天然支持多模态（直接看图），那图片处理流程是否可以简化？

---

## 二、分析结论

图片处理实际分为**两个独立用途**：

| 用途 | 当前 PRD 方案（伪多模态） | Mode D 之后 |
|------|------------------------|-------------|
| **对话中讨论图片** | 先 OCR 提取文字 → 注入 prompt → AI 才"知道"图片内容 | Claude Code/OpenCode 天然多模态，**直接看到图片**，无需预处理 |
| **图片内容进搜索索引** | Vision API → 提取文字 → embedding → LanceDB | **不变** — 仍需提取文字做 embedding，以便未来 RAG 检索 |

**核心结论**：
- **对话理解**：大幅简化。用户贴图后 AI 直接看到并讨论，无需等待 OCR 预处理
- **检索存储**：不变。后台仍需 Vision API 异步提取文字用于向量搜索，但这是后台任务，用户无感
- **这不是额外架构变更**——是 Mode D 架构已经自然带来的附带好处

---

## 三、决策

**从"全链路伪多模态"改为"两层方案"**：
- **第一层 — 对话层**：原生多模态（Mode D 天然支持，零额外开发）
- **第二层 — 检索层**：Vision API 异步提取文字 → embedding → 向量库（后台任务）

---

## 四、PRD 需更新的具体位置

### 4.1 前端架构 → 图片节点支持（PRD 第 236-238 行附近）

**当前写法**：
> V1 伪多模态方案：整图送 LLM Vision API → 文本提取 → embedding

**建议改为**：
> 两层图片处理方案：
> - 对话层：Mode D 框架（Claude Code/OpenCode）天然支持多模态，AI 直接看到图片并理解，无需 OCR 预处理
> - 检索层：后台异步调用 Vision API 提取图片中的文字内容 → embedding → 存入向量库，用于未来 RAG 检索

### 4.2 用户旅程 5 — 图片密集型学习（PRD 第 352-358 行附近）

**当前写法**：
> 节点底部显示"🔄 AI 正在识别图片内容..."，3 秒后变为"✅ 已识别"

**建议改为**：
> 用户贴图后直接点击节点对话，AI 即可看到图片内容并讨论。后台同时显示"📋 正在建立搜索索引..."（异步提取文字用于未来检索），用户无需等待即可开始对话。

### 4.3 FR-KG-06 — OCR 识别（PRD 第 624 行）

**当前写法**：
> 用户粘贴图片到白板时，系统自动异步进行 OCR 识别并提取概念、公式、文字内容

**建议改为**：
> 用户粘贴图片到白板时，系统后台异步提取图片中的概念、公式、文字内容并存入搜索索引。对话中 AI 通过 Mode D 框架原生多模态能力直接理解图片内容，无需等待提取完成。

### 4.4 FR-KG-07 — 状态指示（PRD 第 625 行）

**当前含义**：图片 OCR 处理中→完成（完成前不能讨论）
**更新含义**：搜索索引建立中→完成（对话随时可以进行，索引用于未来检索）
