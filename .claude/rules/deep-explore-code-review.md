# Deep Explore 方法论 + ⛔⛔⛔ 代码质量对抗性审查

## 核心流程

```
1. 深度分析当前代码 — 逐文件理解架构、数据流、已有资产和断裂点
2. ⛔ 代码质量对抗性审查 — 必须启动独立 agent 审查参考代码的质量
3. 社区/论文验证 — 搜索经过社区和论文验证的成熟方案（WebSearch + Context7）
4. 适配匹配 — 将成熟方案与当前代码做具体适配分析
5. 推荐输出 — 列出成熟、高质量、适配当前代码和预期的方案
6. 增量提问 — 逐个主题向用户展示发现，获取方向确认后再深入下一个
```

## ⛔⛔⛔ 代码质量对抗性审查 — 强制执行（最高优先级）

> 当前代码库存在系统性质量问题。任何 deep explore 中参考现有代码时，不可默认代码是正确的。

### 强制规则

**分析/参考/复用现有代码时，必须启动独立的对抗性审查 agent。**

```
Deep explore agent 分析代码 → 发现需要参考/复用的模块
  → ⛔ 启动独立 agent 对抗性审查
  → 产出：质量评级（可复用/需修复/需重写）+ 问题清单
  → 记录 [Code-Review] 到 Graphiti
  → Deep explore agent 基于审查结果调整方案
```

### 审查维度

| 维度 | 检查内容 |
|------|---------|
| **⛔ Mock/假实现** | mock 数据、模拟 API、硬编码假返回值、TODO 占位、签名存在但内部返回空/假数据 |
| **⛔ 管道打通性** | 调用链是否真正连通？接口定义但从未被调用？功能存在但被禁用？ |
| **⛔ 模拟 API** | 端点是否返回真实数据？是否有 `return []` 类占位？ |
| 逻辑正确性 | 算法正确？边界条件？死代码？ |
| 数据流完整性 | 输入输出匹配？数据丢失/变形？ |
| 错误处理 | 异常捕获？降级策略？静默失败？ |
| 性能 | 重复计算？N+1 查询？内存泄漏？ |
| 安全性 | 输入验证？注入？敏感数据？ |
| 可维护性 | 命名？耦合度？硬编码？ |
| 集成完整性 | 模块接口正确？断裂调用链？ |

### ⚠️ 已知系统性问题（2026-03-12 确认）

- Agent API 端点大量 **MOCK 实现**（agents.py 多个端点返回假数据）
- 内存查询 API **全是 TODO**（`return TemporalQueryResponse(items=[], total_count=0)`）
- FSRS、BehaviorTracker、Graphiti Bridge **已实现但从未被调用**
- Phase 3 Agent Graph **完整实现但被禁用**
- Strategy Selector **完整映射但无人调用**
- Verification 系统与 Retrieval 系统**信息断层**
- 前端 `/explain/four-level` 等端点使用 **mock implementations**

### ⛔ 绝对禁止

- ❌ 不审查就说"代码已实现 X%，可以复用"
- ❌ 审查和 deep explore 由同一个 agent 完成
- ❌ 只看函数签名不看实现就下结论
- ❌ 发现问题后隐瞒或轻描淡写
- ❌ 看到函数存在就认为管道打通了

### 其他禁止行为

- ❌ 只看文件名和目录结构就下结论
- ❌ 提出未经社区验证的"创新"方案
- ❌ 一次性 dump 所有发现，不给用户消化和反馈的机会
- ❌ 忽略已有代码资产（很多功能已 60-80% 实现，只需激活/修复）

### 增量提问模式

```
[探索主题 X]
→ 展示代码现状 + 社区最佳实践
→ 给出推荐方案（选最成熟的，不要让用户做微观选型）
→ 问用户："这个方向符合你的预期吗？"
→ 确认后 → 深入下一个子主题
→ 不认可 → 调整方向后重新推荐
```

### 输出标准

每个推荐方案必须包含：
1. **社区验证证据** — 哪些项目/论文已经验证有效
2. **代码适配分析** — 当前代码中哪些可复用，改动量多大
3. **算法原理** — 用用户能理解的方式解释核心逻辑
4. **注意事项** — 可能的坑和规避方法
