## 2. 修复方案 (已落地到 spec)

### V-07 修复: callout 写 Graphiti 必带 wikilink 双链上下文

**文件**: `_bmad-output/implementation-artifacts/epic-1/1-16-callout-graphiti-hook.md`

**关键变更**:
- AC#2 event body 加 5 字段: `node_id` / `source_board` / `out_links[]` / `in_links[]` / `path_trace[]`
- AC#4 episode_body 改自然语言化 (不再 batch json.dumps), 保留 wikilink 关系 + Graphiti embedding 可检索性
- Task 1.5 新加: plugin `wikilink-context.ts` 抓 obsidian `metadataCache.resolvedLinks` / `getBacklinksForFile()` / navigation history
- Task 4: callout_events LanceDB 表 schema 扩 5 字段
- estimate_hours: 3 → **5** (+2h)

**修复理由**: 旧 spec callout episode 只存"句子"不存"探索路径" → 未来 `search_facts("X 和 Y 的关系")` 查不出"用户在原白板 X 沿 A→B→C 探索后, 对 C 写了 tip Y"的完整学习上下文 → 直接违背用户 2026-05-13 核心闭环原话 "**批注+双链探索+个人记忆系统+极其针对性考察**".

### V-08 修复: LITE-4-3 出题路线 0 加 wikilink 邻居

**文件**: `_bmad-output/implementation-artifacts/epic-4/LITE-4-3.md`

**关键变更**:
- AC#1 从 4 路融合升级为 **5 路融合** (新加路线 0: `get_wikilink_neighbors(node_id, hops=2, max_per_hop=5)`)
  - hop 1 邻居附 brief (frontmatter title + body 前 100 字)
  - hop 2 仅 node_id 列表 (避免 token 爆炸)
  - 来源: backend `WikilinkGraphService.get_neighbors()` (复用 STORY-2-10 已建索引)
  - 1s 超时降级
- AC#4 加 "当存在 wikilink 邻居时, 题目需引用**至少 1 个邻居关系**"
- Task 0 新加: WikilinkGraphService.get_neighbors 实现
- Simplification Rules 表: "wikilink 邻居网络 ❌ 禁" → "✅ 保留 (V-08 修复)"
- depends_on 加 `STORY-2-10-wikilink-graphiti-sync` (路线 0 索引必须先建)
- priority 升 P1 → **P0**, sprint Sprint 3+ → **Sprint 2**
- estimate_hours: 3 → **6** (+2h V-08 wikilink, +1h V-10 question_text 持久化, +0h baseline)

**修复理由**: Round-13 Q4 (2026-04-29) 推荐的"三层检索 wikilink 第一层"被 LITE-4-3 v1 直接跳过 → Obsidian 双链只剩 UI 角色 → 出题只考"单节点回忆"不考"概念网络回忆".

### V-10 修复: question_text 持久化 + score_answer 强制回读

**文件**: `_bmad-output/implementation-artifacts/epic-4/LITE-4-3.md` (与 V-08 同 spec)

**关键变更**:
- AC#8 新加: `generate_question` 返回前必须持久化真实 question_text 到 `questions_registry` LanceDB 表 (含 context_used trace)
- AC#9 新加: `score_answer` 必须按 question_id 回读真实 question_text, **禁止**从 `canvas_service.find_node_across_canvases(node_id)` 取 node content 当 question_text. 回读失败抛 `ScoringContextMissingError(422)`, 禁降级
- Task 7 新加: `questions_registry` 表 schema + 删 `exam_tools.py:435-454` 现有 "Story 3.2 fix" 妥协逻辑 + pipeline_token 增加 `question_text_hash` 防篡改

**修复理由**: 现有 `backend/app/mcp/tools/exam_tools.py:435-454` 有显式 comment "Story 3.2 fix: pipeline token contains question_id but not question_text. Fetch concept content from canvas as scoring context." → 这是过去 Story 3.2 时期的妥协, 当时出题不引用 tip 原话, 用 node content 作 scoring context 还合理; 但 LITE-4-3 已能引用 tip 原话 + 修复后还引用 wikilink 邻居 → 评分仍按节点正文评 = **同一段代码, 过去合理, 现在有害**, BKT/FSRS 信号污染从根源开始.

### V-11 修复: LITE-5-6 spec 4 处统一为 dual-write

**文件**: `_bmad-output/implementation-artifacts/epic-5/LITE-5-6.md`

**关键变更**: spec 4 处自相矛盾全部统一为 "frontmatter 同步主写 + Graphiti 异步副写经 STORY-2-10 管道":
- Task 6 "禁 Graphiti 路径" → "异步写 Graphiti 校准事件" (调 `calibration_events` 表, sweep 时 episode_body 自然语言化)
- Simplification Rules 表 "Graphiti add_episode 砍" → "异步保留" + 禁实施清单删去 add_episode
- Dev Notes Architecture "不接 Graphiti" → "dual-write (同步 frontmatter + 异步 calibration_events 经 STORY-2-10 sweep)"
- Automated Checkpoints "禁 Graphiti 调用 grep=0 matches" → 5 个新 checkpoint (events_queue 写入 / sweep 消费 / Graphiti 不可用降级 / 异步语义 grep / e2e Graphiti 闭环)
- depends_on 加 `STORY-2-10-wikilink-graphiti-sync` (共用 events_queue 管道)
- estimate_hours: 2 → **2.5** (+0.5h calibration_events schema + sweep 扩展)

**修复理由**: 旧 spec AC#7 说"双写", 但 Task 6 说"禁", Simplification 表说"砍", Automated Checkpoints grep 检查"0 matches" → dev 接手必按 grep 检查标准实现成 single-write → AC#7 死字 → 校准投票完全无法入 Graphiti 学习历程 → 违背用户 2026-05-13 核心闭环.
