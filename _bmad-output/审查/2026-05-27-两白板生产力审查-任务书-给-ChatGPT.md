# 原白板 + 检验白板生产力标准审查 — 给 ChatGPT 的任务书

## 0. 一句话任务

审查 Canvas Learning System 的**两个核心学习功能**(原白板剖析 + 检验白板 Active Recall), 从**用户实际学习过程需求**出发, 判定它们落地后**是否达到"我愿意每天高质量用它学习"的生产力标准** —— 不是审技术细节, 是审"这两个功能能不能真正支撑一个 CS 61B 学生的日常学习闭环"。

## 1. 用户的真实意图 (锚点, 不可妥协)

### 用户 2026-05-27 原话
> "我们使用 Graphiti 本质上是要我们**原白板和检验白板的功能可以达到生产力标准来高质量使用**, 着重聚焦于我对这两个功能在我实际学习过程的需求是什么。"

**翻译**: Graphiti 是手段不是目的。目的是这两个白板功能本身好用。审查必须站在"用户每天怎么学习"的视角, 不是"代码是否优雅"。

### 用户 2026-05-13 核心闭环原话
> "批注是核心, 我需要用我的个人记忆系统充分理解我使用原白板的学习过程 + 极其针对性的考察。"

### 用户 2026-04-08 锁定 (PRD §2 灵魂)
> "检验白板 100% 等价实现 — 不可妥协。" (Karpicke d=1.50, Canvas 12 个学习设计中效应量最大)

## 2. 两个白板的 PRD 灵魂定义 (真相源)

| 维度 | 原白板 (剖析模式) | 检验白板 (Active Recall) |
|---|---|---|
| **信息可见性** | 信息可见 | **信息隐形** (完全空白 UI, 只看当前题) |
| **学习机制** | 理解 + 剖析 | 主动回忆 (Retrieval Practice) |
| **效应量** | Edge 对话 EI+SE d=0.80-1.00 | **Karpicke d=1.50 (最大)** |
| **用户动作** | 批注 callout / 双链探索 / 派生节点选关系 / Edge 对话 | 答题 (手写) / 4 级提示 / 跳过 / 考后校准投票 |
| **PRD 章节** | §1.1 + §1.2 + §1.3 (line 122-502) | §1.1 + §2 灵魂章节 (line 905-1160) |
| **关键不可破** | wikilink 双链探索是核心 | 三重隔离 (reference_answer=None) + 防嵌套 + 信息隐形 |

## 3. 实际学习过程的 3 条生产力闭环 (审查主线)

### 闭环 A: 原白板日常剖析
```
打开原白板 → Cmd+Shift+A 批注困惑 (callout) → Cmd+Shift+D 派生新概念节点 + 选关系类型 (prerequisite 等)
→ 沿 wikilink 双链探索相邻概念 → Cmd+Shift+C 跟 AI 对话剖析 → 形成认知网络
```
**审查问题**: 这条链路是否流畅到"我愿意每天打开它学习"? 卡点在哪?

### 闭环 B: 检验白板针对性考察
```
Dashboard 一键考察 / Cmd+Shift+Q → 完全空白白板 (信息隐形) → AI 出题 (针对我薄弱点 + 引用我说过的话)
→ 我手写答案 → 静默评分 (AutoSCORE 4维) → 考后校准投票 → 掌握度更新
```
**审查问题**: 出题真针对吗? 信息真隐形吗? 评分可信吗? 掌握度更新闭环吗?

### 闭环 C: 两白板衔接 (学习飞轮)
```
原白板学完 → 检验白板考察 → 暴露新疑问 → 拉出新节点归原白板 → 下次考察能考到这个新疑问
```
**审查问题**: 这个飞轮真闭合吗? 还是断在某一环?

## 4. 实证发现的 5 个生产力风险 (Claude 已查, ChatGPT 深审)

> 这些是 2 个并行 Explore agent 实证 codebase 后发现的, ChatGPT 应评估它们对**生产力标准**的实际影响。

### 风险 P-1: 检验白板"灵魂"在代码中几乎无踪 (最严重)
- PRD §2 锁定的 `/start_exam_board` 10 步 workflow + `type: exam_board` 防嵌套 frontmatter, 在代码中**只有 1 句 UI 文案** (main.ts:1602), 无路径/frontmatter 写入逻辑
- 实际落地的是 **MVP-α 简化版** (exam-quick.ts → `节点/考察-{concept}-{date}.md`), 不是 PRD 的"100% 等价实现"
- **生产力影响**: 用户 2026-04-08 锁定"不可妥协", 但当前是简化版。差距有多大? 简化版够不够"高质量使用"? 还是必须补回灵魂机制?

### 风险 P-2: 三套路径命名互不一致
- PRD: `exam_boards/<slug>-<ts>.md` + `wiki/concepts/`
- CLAUDE.md round-11 扁平架构: `检验白板/<exam>.md` + `原白板/` + `节点/`
- 实际代码: `原白板/` (main.ts:1466 校验) + `节点/考察-{concept}-{date}.md` (exam-quick.ts:47)
- **生产力影响**: 用户实际用的时候, 考察文件到底存哪? 检验白板和节点混在 `节点/` 目录? 这影响用户能否找到/复习历史考察

### 风险 P-3: 三重隔离是赋值级非 gate 级 (检验白板灵魂红线)
- `reference_answer=None` 在 exam_tools.py:300/305 是赋值, 但**无 fail-closed gate** 强制清除
- 全库 `fail-closed` 只在 API auth, **检验白板评分/隔离链零 fail-closed**
- **生产力影响**: 如果隔离被破 (answer 泄漏给用户), Karpicke d=1.50 直接崩塌。这是灵魂红线, 赋值级够安全吗?

### 风险 P-4: 两条评分链并存 (V-10 真相不唯一)
- MCP 链 (exam_tools.py:435-454): 评分用 node content 当题面 → **V-10 漏洞**
- MVP 链 (exam_grade.py:93): 读 `record["question_text"]` 真实题面 → **已规避 V-10**
- **生产力影响**: 两条链哪条是真相? 用户实际走哪条? 掌握度信号会不会因为走错链被污染?

### 风险 P-5: 原白板上下文富集读 .canvas JSON (降级后跑不通)
- context_enrichment_service.py:404 还读 `.canvas` JSON, 但扁平架构没有 .canvas 文件
- epics.md line 161 AR11 已承认"需重构为 wikilink 图遍历", 但 spec 还 ready-for-dev 未实施
- **生产力影响**: 原白板 AI 对话/出题时拉邻居上下文, 在扁平架构下实际跑通了吗? 还是跑偏/跑空?

## 5. 7 个审查议题 (希望 ChatGPT 给具体判定)

### 议题 1: 检验白板 PRD 灵魂 vs MVP 简化版差距
当前 MVP-α 简化版 (exam-quick.ts) 离 PRD "100% 等价实现" 差多少? 哪些灵魂机制 (防嵌套/隔离 gate/10步 workflow/信息隐形) 是**生产力必须补**, 哪些可以简化接受?

### 议题 2: 原白板剖析闭环 (闭环 A) 生产力评分
批注 + 双链探索 + 派生节点 + Edge 对话 这条链路, 按"我愿意每天用"标准打分 (1-10). 卡点排序。

### 议题 3: 检验白板考察闭环 (闭环 B) 生产力评分
出题针对性 + 信息隐形 + 评分可信 + 掌握度更新, 按生产力标准打分。V-08 (出题绕开 wikilink) + V-10 (评分对象漂移) 对生产力的实际影响。

### 议题 4: 学习飞轮 (闭环 C) 是否闭合
原白板 → 考察 → 新疑问 → 归原白板 → 下次考到。这个飞轮当前断在哪一环? 补哪个 spec 能闭合?

### 议题 5: 三套路径命名收敛方案
PRD / CLAUDE.md / 代码三套路径, 推荐统一到哪套? 迁移成本? 对用户"找历史考察"的影响?

### 议题 6: "实现领先规格" 债务收敛
代码已扁平架构, 但 PRD 30+ 处 + Epic 2/3/4/5 spec + 后端 context_enrichment 还旧路径。这个"实现领先规格"的债怎么收敛? 是改 spec 追代码, 还是反过来?

### 议题 7: Graphiti runtime (5-ge 主干) 与两白板生产力的关系
我已规划 5 个 Graphiti spec (统一 schema + belief 版本链 + facade)。它们是两白板的"记忆地基"。审查: 地基做完后, 两白板生产力自动达标吗? 还是两白板本身还有独立的 UX/出题质量 spec 要补?
