## 3. 三个真实缺口（全部 file:line 实证）

### 🔴 GAP-A：S2-2 belief 版本链通过主管道**永远不会触发**（死链）

- `memory_service._enqueue_episode`（`memory_service.py:310-343`）构造 `EpisodeTask` 时**没传 `metadata`** → `task.metadata` 永远空 `{}`
- 我的 belief hook（`episode_worker.py:569`）门控 `task.metadata.get("event_type") in EVOLUTION_EVENT_TYPES` → 永远 `None` → **永不触发**
- 且现有生产方发的 event_type 是 `"learning_tip"`/`"misconception"`/`"calibration"`/`"exam_node_created"`（自由文本），**没有一个**用 S2-2 新枚举（`callout_updated` 等）
- → belief 链代码正确且测试通过，但**是座孤岛**，没有生产路径喂它

### 🟡 GAP-B：统一 `CanvasGraphEpisodeV1` schema 无人采用

- C-1"唯一写入契约"定义了，但所有生产方仍传纯文本 content + 自由 event_type，没走统一 schema
- 证据：`canvas_episode.py` 的 `callout_added`/`wikilink_added` 字面量只在自己文件 + narrative 示例里出现（`grep` 全库零生产方）

### 🔴 GAP-D（验证方向1 时新发现，主链断点）：累积批注可能根本到不了检验白板

- `_get_tips` 查询要求**同一个** `EpisodicNode` 同时有 `source_description IN [canonical]` **且** 顶层 `e.node_id`（`question_generator.py:903-910`）
- 但写入端两条路各缺一半：
  - `neo4j.record_episode`（`neo4j_client.py:731-765`）**根本不建 EpisodicNode**，委托 `create_learning_relationship` 建 `(:User)-[:LEARNED]->(:Concept)` — 与 `_get_tips` 无关
  - Graphiti `add_episode` 路径（`_enqueue_episode`）建 EpisodicNode 且有 `source_description`，但**不传 node_id** → EpisodicNode 大概率无顶层 `node_id` 属性
- → `_get_tips` 的 `AND e.node_id = $node_id` 很可能**永远 0 命中** → 检验白板看不到累积批注
- ⚠️ 静态推断, 需连真实 Neo4j 实测确认（Graphiti 是否经 episode_body 间接存 node_id 不确定）
- 历史佐证: 同文件注释（`memory_service.py:1270-1274`）记载 P0-2a 已修过一次同类 reader/writer 不对齐（source_description），node_id 对齐可能仍是漏网

### 🟡 GAP-C：写入与读取是**两套节点命名空间**，不连通

- 检验白板**读**：`EpisodicNode WHERE node_id`（批注）+ `CanvasNode-[CANVAS_EDGE]->CanvasNode`（原因边）
- S2-2 belief**写**：`Entity-[RELATES_TO {belief_key}]->Entity`（uuid5 节点，按 node_id 命名，独立命名空间）
- → belief 时序链即使触发，**检验白板也读不到**（代码审查 M2 已预警，现确认对读取端有实际影响）
