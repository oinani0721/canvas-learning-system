## 调研主题 1：评分权重设计是否合理

### 问题

PRD 截图展示"应用能力 40%"由 4 个信号加权：考察评分(40%) + 错误频率(25%) + 对话表现(20%) + 自评(15%)。这个设计是否有学术支撑？

### 代码审查结论 [Code-Review]

| 模块 | 状态 | 评级 |
|------|------|------|
| BKT+FSRS 单值精通度 (mastery_engine.py, 471行) | 真实实现，数学正确 | **可复用** |
| Beta-Bayesian 五维融合 | **零代码** | 需从零开发 |
| 10 信号→5 维映射 | **零代码** | 需从零开发 |
| 考察评分→精通度管道 | verification_service 不调用 MasteryEngine | **管道断裂** |
| 错误频率采集 | BehaviorTracker 存在但从未被后端调用 | **管道断裂** |
| Calibration Tracking | **零代码** | 需从零开发 |
| OLM 三层展示 | 仅 Layer 1 (MasteryDashboardView) | Layer 2+3 缺失 |

### 学术调研结论 [Research-Tech]

**固定权重 vs 动态权重**：
- 无直接 A/B 论文，但 BKT→DKT 演进方向明确倾向动态
- FINCH/DyLam 框架实验证明动态优于固定
- 固定权重作为 MVP 起点合理，条件是预留动态调整接口

**自评可靠性**（meta-analysis 数据）：
- 无反馈时 r = 0.19~0.39（几乎不可靠）
- 有反馈时 r = 0.51（勉强可用）
- 15% 权重方向正确，但更好的方式是 Area9 式盲区探测

**LLM 评分可靠性**：
- 最佳场景（精心 prompt + rubric + 校准样本）：QWK 0.72-0.80, kappa > 0.80
- 一般场景：kappa 0.53-0.70
- 需先通过可靠性验证再启用

**竞品权重方案**：
| 产品 | 信号来源 | 用自评吗 |
|------|---------|---------|
| ALEKS | 100% 答题 | ❌ |
| Khan Academy | 100% 答题 | ❌ |
| Duolingo | 100% 答题 | ❌ |
| Knewton | 答题 + 时间衰减 | ❌ |
| Area9 Lyceum | 答题 + 4级置信度 | ✅ 仅作盲区探测 |

### 修正建议（待用户确认）

1. 权重展示从固定 40/25/20/15 改为标注"等效权重（动态计算）"
2. "上次应用题得 35/100" 改为"历次应用考察综合评估"
3. "应用类错误 3 次" 改为"最近 N 次中错误率 X%"
4. AI 评分增加"需先通过可靠性验证才启用"说明
5. 自评从"加权分量"改为 Area9 式"盲区探测器"

## 调研主题 2：OLM 展示对用户的实际价值

### 用户反馈

> "我看这个数字时我对它的感知性不强" — 产品 owner 直接反馈

### 学术调研结论 [Research-Tech]

**LAK'24 系统综述（38 项研究）**：
- 仅展示描述性数据的学习分析仪表板，**仅 25% 有正面效果**
- 处方性展示（告诉你该做什么）效应量 ES=1.36（非常强）

**Bull & Kay OLM 30 年研究**：
- 白箱 OLM 价值不是"让学生看分数"，而是触发反思/规划/自我监控/信任/模型修正
- **可协商 OLM > 可查看 OLM** — 让学生挑战模型比纯展示更有效

**竞品展示方式**：
- Khan Academy: 4 级离散等级，不用百分比
- Duolingo: 连胜天数 + 皇冠等级，不用百分比
- Anki: 复习预测（今天复习几张），不用百分比
- **无成功产品展示权重百分比**

### 重设计方向（待用户确认）

Layer 2 从"算法解释"转向"诊断+处方+趋势"：
- 去掉权重百分比展示
- 数字变事实描述
- 加趋势指示（比上周进步了）
- 加行动建议（开始练习按钮）
- 加可协商入口（不同意→检验白板验证）

### Pencil 可视化

已在 `UI 相关设计样式.pen` 中创建 Before/After 对比 + 4 步可协商工作流（frame id: qcm7j）

## 调研主题 3：UI 设计是否符合产品实际

### 用户反馈

> "你这里的UI设计就是废物，脱离了我们的实际" — 产品 owner 直接批评

### 代码审查结论 [Code-Review]

| 触点 | 有精通度信息？ | 状态 |
|------|-------------|------|
| Canvas 节点本身 | ❌ 零视觉指示 | 未实现 |
| 右键菜单 | ❌ 无 mastery 入口 | 未实现 |
| 对话面板 | ❌ 对话后无反馈 | 未实现 |
| 检验白板 Modal | ✅ 实时精通度% | 已实现 |
| 节点颜色变更 | ✅ 自动触发 self-assess | 已实现 |
| 右侧栏 Dashboard | ✅ MasteryDashboardView (440行) | 已实现（需主动打开） |

### 核心结论

**精通度展示面板不是刚需。** 理由：
1. 用户核心成功标志是"系统越来越懂你" = Agent 记忆检索质量，非看面板
2. MVP 核心 6 项第 6 项是"精通度更新**通知**"而非"展示面板"
3. 精通度数据的主要消费者是系统自身（BKT 选题/FSRS 调度），非用户
4. 独立面板脱离 Obsidian 插件的实际交互模式

### 建议

精通度展示面板从 Phase 1 降级至 Phase 2+。精通度信息改为融入已有交互流：
- 节点颜色（被动指示）
- 考察结束后一句话通知
- FSRS 复习提醒

## 调研主题 4：MVP Dashboard 刚需定义

### 用户定义

> "目前我最小MVP，要的 Dashboard 就是开启原白板，开启原白板的检验白板，检验白板的考察模式"

### PRD 覆盖情况

- 用户旅程 2（第 318 行）描述了完整流程：Dashboard→选择原白板→生成检验白板→考察
- **但 FR 章节无任何一条定义 Dashboard 作为操作枢纽**
- PRD Gap：旅程描述了流程但功能需求漏了定义

### 代码审查结论 [Code-Review]

| 功能 | 代码状态 | 问题 |
|------|---------|------|
| 开原白板 | ❌ 没有白板列表 | 只有复习任务列表，无法浏览选择白板。"开始复习"是 TODO |
| 开检验白板 | ❌ 按钮坏了 | getCurrentCanvasPath() 在 Dashboard 活动时返回 null |
| 启动考察 | ❌ 零 UI 入口 | 考察功能真实但仅在右键生成检验白板后自动触发 |

**后端全部真实可用**，前端管道全断。

### 建议

1. PRD 补充 Dashboard 相关 FR（白板浏览/检验白板生成/考察启动）
2. 当前 3486 行 ReviewDashboardView 过度设计，MVP 只需：
   - 原白板列表 + [打开]
   - 检验白板列表 + [打开] + [开始考察]
   - [为某白板生成新检验白板]
3. 统计图表/精通度分布等降级为 Phase 2
