## 二、Phase 1 — 约束地图（7 条约束 + 5 层优先级）

### 2.1 验证优先级排序

| 层级 | 验什么 | 为什么这个顺序 |
|------|--------|---------------|
| 🥇 地基 | 系统配置（S4）+ 内容切片（A1） | 全部功能的前提，不通则全部白测 |
| 🥈 核心 | 搜索范围（A3）→ 搜索排序（A2） | 用户每次搜索都直接感受；范围先于排序 |
| 🥉 支撑 | 笔记刷新（A4） | 改笔记后才暴露；与 A3 有暗链 |
| 4️⃣ 增强 | 截图搜索（A5-图片） | 日常学习常用 |
| 5️⃣ 延后 | PDF 搜索（A5-PDF） | 偶尔使用，优先级最低 |

### 2.2 约束清单

| # | 约束内容 | 来源 |
|---|---------|------|
| 1 | S4 配置 + A1 切片是最高优先级（地基） | 用户直觉确认 + 技术依赖分析 |
| 2 | A5-图片有优先级，A5-PDF 降低 | 用户反馈：图片日常用，PDF 偶尔用 |
| 3 | A2 搜索排序是"即时体感"，优先级仅次于地基 | 用户每次搜索都能直接感知 |
| 4 | A3 范围控制是"安全网"，适合批量自动化测试 | 属于"暗伤"，不易人工逐条发现 |
| 5 | A4 刷新与 A3 有暗链（~~非独立~~） | 代码审查发现 SubjectResolver 链条 |
| 6 | A2 和 A3 有暗依赖（范围影响排序质量） | 范围太宽 → 无关结果稀释排序 |
| 7 | 图片和 PDF 可拆分验证 | 用户需求不同、管道部分独立 |

### 2.3 深度探索发现（8 个隐藏依赖）

| # | 隐藏依赖 | 影响 |
|---|---------|------|
| 1 | **3 套独立配置系统** + 2 个隐藏配置文件 | S4 修复范围比预想大（5 个源全部需统一） |
| 2 | **向量维度硬编码冲突**（768/384/1024 三者不一致） | A1 换模型必须同步更新多处硬编码 |
| 3 | **5 个 LanceDB 单例客户端** | 不同模块用不同配置访问同一数据库 |
| 4 | **A3→A4→A2 链条**（非直接 A3→A2） | A3 范围判定 → A4 刷新时打标签 → A2 搜索时用标签过滤 |
| 5 | **隐藏后处理重排**（reference_config.py） | 搜索排序后还有一个配置文件偷偷调整排名 |
| 6 | **多模态和文本共享 LanceDB 路径** | 并发写入可能文件锁竞争 |
| 7 | **RAG Service sys.path 污染** | 影响测试隔离性 |
| 8 | **YAML 配置未纳入统一管理** | subject_mapping.yaml 独立于 Settings 系统 |

### 2.4 修正后约束地图

```
     🔧 系统配置（⚠️ 5 个配置源全部需统一）
                    ↓
     🧱 笔记切片质量
                    ↓
          ┌─────────┴─────────┐
          ↓                   ↓
   🎯 搜索范围 ──────→ 🔄 笔记刷新
  （范围判定）       （刷新时用范围的标签）
          │                   │
          └────────┬──────────┘
                   ↓
          🔍 搜索排序 + 🕵️ 隐藏裁判
          （两者一起验才准）
                   ↓
          🖼️ 截图搜索（⚠️ 和文字共用仓库）
                   ↓
          📄 PDF 搜索（最后）
```

## 三、Phase 2 — 形态分析（4 维度测试组合矩阵）

### 3.1 维度 1：测试素材策略

| 素材 | 用途 | 复用次数 |
|------|------|---------|
| 📝 10 个标准问题包（从 60 题路由 benchmark 精选） | 搜索排序 + 搜索范围 + 刷新验证 | 一套测三层 |
| 📄 1 篇"刁钻"笔记（中英混合 + LaTeX + 代码块） | 切片质量专项检查 | 1 次 |
| 🖼️ 5 张 CS188 课件截图 | 截图搜索验证 | 1 次 |

**核心发现：一套素材多用，节省约 40% 测试准备工作量。**

### 3.2 维度 2：测试方式

| 验证层 | 方式 | 理由 |
|-------|------|------|
| 配置（5 源） | 自动化脚本 | 确定性检查，脚本一跑就知道 |
| 切片质量 | 自动化统计 + 人工抽检 5 个样本 | 自动化抓大面，人工看细节 |
| 范围+刷新 | 自动化集成测试 | 端到端链路必须整体跑 |
| 排序+隐藏裁判 | A/B 对比（新旧结果人工判断） | 排序好坏只有人能判 |
| 截图搜索 | 人工验证 | 截图内容对不对必须用眼睛看 |

### 3.3 维度 3：测试分组

| 批次 | 包含内容 | 覆盖 DR 数 | 门禁类型 |
|------|---------|-----------|---------|
| A（地基） | 配置 5 源统一 + 切片质量 | ~5 条 | 硬门禁 |
| B（核心） | 范围→刷新→排序→隐藏裁判，10 题端到端 | ~10 条 | 硬+软门禁 |
| C（增强） | 截图搜索 | ~3 条 | 软门禁 |
| D（延后） | PDF 搜索 | ~1 条 | 可选 |

### 3.4 维度 4：判定标准

| 层级 | 合格标准 | 不合格信号 |
|------|---------|-----------|
| 配置 | 5 个源返回同样的值 | 任何一个不一致 |
| 切片 | 5 个样本无断句/公式完整 | 出现断句或公式被切开 |
| 范围 | 搜 CS188 只出 CS188 结果 | 出现其他课内容 |
| 排序 | 10 题中 ≥7 题第一条结果最相关 | <7 题 |
| 截图 | ≥3/5 截图可搜到且内容正确 | <3/5 |

### 3.5 深度探索发现 — 测试基础设施审计

**代码审查结论：方案方向正确，但验房员手上没有工具。**

| 状态 | 项目 | 影响 |
|------|------|------|
| ❌ 缺失 | Ground Truth Q&A 数据集（标准答案表） | 批次 B 无法执行 |
| ❌ 缺失 | 评估框架（RAGAS/DeepEval） | 无自动化质量度量 |
| ❌ 缺失 | 端到端 RAG 测试脚本 | 无法自动化跑 10 题 |
| ❌ 缺失 | 配置对比脚本 | 批次 A 需人工逐个翻 |
| ✅ 可用 | evaluator.py MRR@K 计算器 | 真实实现，可直接复用 |
| ✅ 可用 | 60 题路由 benchmark | 可做标准问题的原料 |
| ✅ 可用 | LanceDB 真实索引数据 | 之前跑过索引，可直接搜索 |

**新增 Phase 0：搭建工具箱（预估 10-14 小时）**

### 3.6 社区最佳实践验证

| 我们的方案 | 社区共识 | 匹配度 |
|-----------|---------|--------|
| 一套素材多用 | Golden Dataset 模式 — 业界标准 | ✅ 完全吻合 |
| 自动化打底 + 人工精检 | 分离检索/生成测试 — 业界共识 | ✅ 完全吻合 |
| 批次递进门禁 | RAGOps 分层验证 — 论文支持 | ✅ 完全吻合 |
| A/B 对比排序 | Interleaving 模式更灵敏 — 可采用 | ✅ 可增强 |
| 推荐框架 | DeepEval (Pytest 原生 CI/CD) | ✅ 最成熟 |

## 四、Phase 3 — 决策树映射（执行路径 + 应急预案）

### 4.1 完整执行路径

```
Phase 0: 搭工具箱 (~10-14h)
├── 从 60 题中精选 10 个标准问题 + 标注"正确答案应包含哪些笔记"
├── 安装 DeepEval 评估框架
└── 写 3 个脚本（配置对比 / 切片统计 / 端到端测试）
│
↓ 工具箱就绪
│
批次 A: 地基检查
├── 5 个配置源是否统一？
├── 切片质量抽检（5 个样本）
├── ✅ 全过 → 进入批次 B
├── ⚠️ 1-2 个源不一致 → 继续但标注"前提不满足"（软门禁）
└── ❌ 完全不通 / 切片严重断裂 → ⛔ 停！修地基
│
↓
│
批次 B: 核心链路（10 题端到端）
├── 搜索范围对不对？
├── 刷新后能搜到新内容？
├── 排序质量（含隐藏裁判）
├── ✅ ≥7/10 → 进入批次 C
├── ⚠️ 5-6/10 → 记录问题，继续测 C
└── ❌ <5/10 → ⛔ 停！分析根因
│
↓
│
批次 C: 截图搜索（5 张截图）
├── 搜相关问题能找到截图？
├── 截图内容识别正确？
├── ✅ ≥3/5 → 批次 D（可选）
└── ❌ <3/5 → 记录问题，不阻断主流程
│
↓
│
总体报告
├── 每条 DR: PASSED / FAILED / BLOCKED
├── 发现的问题清单
└── 建议的修复优先级
```

### 4.2 应急预案

| 如果… | 怎么办 | 严重程度 |
|-------|--------|---------|
| 配置完全不通 | 停止一切验证，S4 修复未完成 | 🔴 阻断 |
| 切片把句子切断 | 回去修 A1 分块策略 | 🔴 阻断 |
| 搜索跑到别的课 | A3 范围控制 bug，修复后重测 | 🟡 修复后继续 |
| 改了笔记搜不到 | A4 刷新机制失效，不阻断其他测试 | 🟡 记录继续 |
| 排序不准（<5/10） | 分析根因（范围？裁判？模型？），定点修复 | 🔴 阻断 |
| 截图搜不到 | 增强功能缺失，不影响核心 | 🟢 记录待修 |
| 工具箱搭建卡住 | 用现有 evaluator.py 做简化版测试 | 🟡 降级方案 |
